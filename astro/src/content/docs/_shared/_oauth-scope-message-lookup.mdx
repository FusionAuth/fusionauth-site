OAuth scope consent messages and details follow an ordered theme message lookup policy from most- to least-specific. The theme will render the first option that contains a value using the `OAuth2 consent` template's `resolveScopeMessaging` function. The following examples are provided in order from most preferred to least preferred using the `data:read` scope name as a demonstration. Note that a `:` character in the scope name needs to be escaped in the theme messages definition.

```properties
# 1) Override at the application level using the application Id
[{application}a2afb0f5-eb2d-4d8f-a55d-05e978e95fda]{scope-message}data\:read=Application-specific consent message
[{application}a2afb0f5-eb2d-4d8f-a55d-05e978e95fda]{scope-detail}data\:read=Application-specific consent detail

# 2) Override at the tenant level using the tenant Id
[{tenant}e122574f-6ec5-4399-b6b2-04ede9796380]{scope-message}data\:read=Tenant-specific consent message
[{tenant}e122574f-6ec5-4399-b6b2-04ede9796380]{scope-detail}data\:read=Tenant-specific consent detail

# 3) Override at the theme level
{scope-message}data\:read=Themed consent message
{scope-detail}data\:read=Themed consent detail
```

If a themed message cannot be found using the given application, tenant, or theme, the <InlineField>scope.defaultConsentMessage</InlineField> and <InlineField>scope.defaultConsentDetail</InlineField> values will be used. If <InlineField>scope.defaultConsentMessage</InlineField> is empty, the <InlineField>scope.name</InlineField> will be used as the message. If <InlineField>scope.defaultConsentDetail</InlineField> is empty, no detail value will be displayed.
