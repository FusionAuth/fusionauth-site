Building a production Model Context Protocol (MCP) server requires you to consider not just the AI-powered tools and services you expose, but also how to control and track which clients can access them. With FusionAuth, you can add OAuth protection to your MCP server, letting you authenticate clients and enforce per-tool scopes, while maintaining full control with a self-hosted Identity Provider.

Starting with a simple, unprotected MCP server and adding OAuth protection step by step, this guide demonstrates how to:

* Examine a basic MCP server with a simple `get_name` tool.
* Add FusionAuth OAuth configuration and token validation.
* Register an MCP client (such as Claude Desktop) and test the protected tool.

The following tutorial builds a working example of an OAuth-protected MCP server that you can extend with additional tools and security policies.

## Prerequisites

To follow this guide, you need the following installed:

* Docker Desktop (ensure it's running).
* An MCP client such as Claude Desktop or Cursor.
* Python 3.12 or later (and `pip`).
* Node.js version 20.18.1 or later (ensure it's available in your PATH).
* FusionAuth 1.63.0 or later. The Docker stack in this guide uses the `latest` image tag, which satisfies this requirement. If you are running FusionAuth separately, version 1.63.0 or later is required because earlier versions do not advertise S256 PKCE support in the OIDC discovery document, which `mcp-remote` requires.

You also need a **FusionAuth Essentials license** (or higher). Custom OAuth scopes require an Essentials license. If you already have a paid plan, you can find your license key at [account.fusionauth.io](https://account.fusionauth.io/account/plan/). Otherwise, [contact FusionAuth](/pricing) to obtain one.

## Adding OAuth Protection

Now, let's add authentication piece by piece to protect the MCP server, starting with configuration and ending with a fully protected server.

### OAuth Protection For MCP Servers

To protect an MCP server, you need to add:

1. **Token Validation:** When a client sends a request, you need to verify that its access token is valid.
2. **Scope Enforcement:** MCP clients require specific scopes (permissions) to execute tools.
3. **User Context:** Once a request has been authenticated, you use the token to identify which user made the request.

This guide uses FusionAuth's JWT validation endpoint for token validation. It verifies the token's signature using FusionAuth's public keys and returns the token claims, including scopes, if the token is valid.

## Registering MCP Clients As OAuth Clients

Before MCP clients like Claude Desktop or Cursor can connect to your protected MCP server, you need to register them as OAuth clients in FusionAuth.

### The Registration Process

Each MCP client needs its own dedicated OAuth application in FusionAuth. This means if you want to support both Claude Desktop and Cursor, you register two separate applications, one per client. When an MCP client initiates the OAuth grant, it looks like this:

1. The MCP client redirects the user to FusionAuth to log in.
2. The user authenticates their account.
3. FusionAuth redirects the user back to the MCP client with an authorization code.
4. The MCP client exchanges the code for an access token.
5. The MCP client passes the token to the MCP server with each request.

For the OAuth grant to work, FusionAuth needs to know:

* Which redirect URLs are allowed for each MCP client.
* Whether PKCE is required (some MCP clients use PKCE for security).
* Which OAuth grants are enabled.

### What The Setup Script Does

The `setup_clients.py` script automates this registration process. It is located in the `setup` directory. Let's look at the key parts.

The script prompts you for a name for the client, then creates an OAuth application in FusionAuth using the FusionAuth Python client. Run it once for each MCP client you want to register. It registers exact redirect URLs for a fixed callback port (default `3334`):

<RemoteCode url={frontmatter.codeRoot + "/protected-local-mcp/setup/setup_clients.py"} lang="python" tags="create-client-application"/>

This includes the following key configuration values:

* `authorizedURLValidationPolicy: "ExactMatch"` requires the redirect URL to match one of the registered URLs exactly. This is more secure than wildcard matching. Pass `--port` to the setup script if port `3334` is already in use on your machine.
* `proofKeyForCodeExchangePolicy: "Required"` enforces PKCE for security.
* `clientAuthenticationPolicy: "NotRequiredWhenUsingPKCE"` means that no client secret is needed when PKCE is used.
* `scopeHandlingPolicy: "Strict"` only allows scopes that are explicitly registered on the application.

The script returns the generated client Id and the full MCP configuration block to paste into your client config.

## Configuring Your MCP Client

Now let's configure the MCP client to connect to the OAuth-protected MCP server. This section covers the config file approach used by most MCP clients. CLI tools with native HTTP transport support are covered in the [Deploying A Remote MCP Server](#deploying-a-remote-mcp-server) section.

### Configuration Format

Config-file MCP clients use the following configuration structure:

```json
{
  "mcpServers": {
    "name-server": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "http://localhost:8000/mcp",
        "3334",
        "--allow-http",
        "--static-oauth-client-info",
        "{\"client_id\":\"YOUR_CLIENT_ID_HERE\"}"
      ]
    }
  }
}
```

Replace `YOUR_CLIENT_ID_HERE` with the client Id output by the registration script for this client.

Each parameter does the following:

* `command` uses `npx` to run the `mcp-remote` bridge tool.
* `mcp-remote` bridges MCP clients (which expect stdio) to your HTTP-based MCP server. MCP client config files only support stdio servers, so `mcp-remote` is required to connect to any HTTP-based MCP server, regardless of whether auth is involved.
* `--allow-http` allows non-HTTPS connections for local development.
* `--static-oauth-client-info` provides the preregistered client Id from the setup script. This is required because FusionAuth does not support Dynamic Client Registration (DCR). Native MCP clients that attempt DCR will fail to connect. By passing a pre-registered client Id, `mcp-remote` bypasses DCR entirely.

## How It Works

Now that you've built and tested the OAuth-protected MCP server, let's review the key concepts.

### The OAuth Flow

When an MCP client connects to the MCP server, here's what happens:

1. **Discovery:** The MCP client discovers OAuth endpoints from FusionAuth's OpenID Connect configuration.
2. **Authorization:** The MCP client redirects the user's browser to FusionAuth's login page.
3. **Authentication:** The user logs in and is redirected back to the MCP client.
4. **Token exchange:** FusionAuth returns an authorization code, which the MCP client exchanges for an access token.
5. **API calls:** The MCP client includes the access token in requests to the MCP server.
6. **Validation:** The MCP server validates tokens using FusionAuth's JWT validation endpoint before executing tools.

### Token Validation Approach

The MCP server validates tokens using the FusionAuth Python client's `validate_jwt` method, which calls FusionAuth's `/api/jwt/validate` endpoint. This endpoint:

* Verifies the JWT signature using FusionAuth's public keys.
* Confirms the token has not expired.
* Returns the token claims, including the `scope` field, if the token is valid.
* Does not require client credentials.

<Aside type="note">
If you try to use FusionAuth's introspect endpoint here, it will return a 401. Introspection requires that the credentials used to call the endpoint belong to the same application that issued the token. Because the MCP server is a separate FusionAuth application from each MCP client, it cannot introspect the client's tokens. See [FusionAuth issue #3010](https://github.com/FusionAuth/fusionauth-issues/issues/3010) for details. The `/api/jwt/validate` endpoint does not have this restriction.
</Aside>

### Scope Enforcement

The `required_scopes=["get_name"]` parameter ensures that:

* Only tokens containing the `get_name` scope can execute tools.
* FastMCP automatically validates scopes before tool execution.
* Users must be registered to an application with this scope to obtain a token containing it.

### Custom Scopes In FusionAuth

FusionAuth requires an Essentials license (or higher) to use custom scopes. The setup script creates the `get_name` scope on each client application it registers. When users authorize the application, FusionAuth redirects them back to the MCP client with an "Authorization successful!" message.

## Production Considerations

This example demonstrates the core patterns for OAuth-protected MCP servers. For production deployments, consider including:

* **HTTPS:** Use HTTPS for all endpoints, remove the `--allow-http` flag, and configure proper TLS certificates.
* **Token caching:** Cache token validation responses to avoid calling FusionAuth's JWT validation endpoint on every request. Set the cache TTL to match token expiration.
* **Refresh tokens:** Configure your FusionAuth application to issue refresh tokens for long-lived sessions. `mcp-remote` stores tokens in `~/.mcp-auth` and handles token refresh automatically. Support for the refresh grant varies across other MCP clients, so check your client's documentation.
* **Multiple scopes:** As you add more tools, create specific scopes for each tool and configure the token verifier to require the appropriate scopes.
* **Monitoring:** Log token validations and tool calls in the MCP server. In FusionAuth, use the [login records API](/docs/apis/login#search-login-records) to track which clients have authenticated and when.
* **Rate limiting:** Implement per-user rate limits and return HTTP `429` responses when limits are exceeded.
* **Stale client cleanup:** Use FusionAuth's [login records API](/docs/apis/login#search-login-records) to identify client applications that haven't authenticated recently, then deactivate or delete them to keep your application list clean.
* **Preregistering clients at scale:** If you need to onboard many MCP clients, automate registration using the FusionAuth Application API rather than running the setup script manually for each one.

## Deploying A Remote MCP Server

For a production deployment, you would host the MCP server on a remote machine and connect to it over HTTPS. The server code requires no changes. The only differences are infrastructure, configuration, and how MCP clients connect.

### Infrastructure Requirements

For a remote deployment you need:

* A server hosting your MCP server, accessible over the internet.
* A reverse proxy such as [Caddy](https://caddyserver.com/) (which handles TLS certificates automatically) or TLS configured directly on your load balancer to serve HTTPS on the MCP server endpoint.
* A publicly accessible FusionAuth instance, since the OAuth browser redirect requires the user's browser to reach FusionAuth.

### Server Configuration

The [`protected-remote-mcp`](https://github.com/fusionauth/fusionauth-protected-mcp-server/tree/main/protected-remote-mcp) folder of the example repository contains a `docker-compose.yml` that runs only the MCP server, with no FusionAuth or database. Copy it to your server and set the following environment variables when starting:

* `MCP_SERVER_URL`. Set this to your server's public HTTPS URL (e.g. `https://mcp.example.com`). FastMCP uses this to construct the OAuth metadata that MCP clients discover.
* `FUSIONAUTH_URL`. Set this to your FusionAuth instance's public URL. The MCP server uses this to validate tokens internally.
* `FUSIONAUTH_EXTERNAL_URL`. Set this to the same value. This is the URL that MCP clients use in their browser to complete the OAuth flow. It is kept separate so that in containerized deployments where FusionAuth is reachable internally at a different address, the two can differ.

```bash
MCP_SERVER_URL=https://mcp.example.com \
FUSIONAUTH_URL=https://auth.example.com \
FUSIONAUTH_EXTERNAL_URL=https://auth.example.com \
docker compose up -d
```

### Registering Remote MCP Clients

Run the setup script from the `protected-remote-mcp/setup` directory, passing `--fusionauth-url`, `--api-key`, and `--mcp-server-url` to point it at your remote FusionAuth instance and MCP server:

```bash
python3 setup_clients.py \
  --fusionauth-url https://auth.example.com \
  --api-key YOUR_API_KEY \
  --mcp-server-url https://mcp.example.com
```

If your FusionAuth instance has multiple tenants, also pass `--tenant-id` to target the correct one:

```bash
python3 setup_clients.py \
  --fusionauth-url https://auth.example.com \
  --api-key YOUR_API_KEY \
  --mcp-server-url https://mcp.example.com \
  --tenant-id YOUR_TENANT_ID
```

<Aside type="note">
Use `claude mcp add-json` with the config output by the setup script to add the server in Claude Code.
</Aside>

### Connecting MCP Clients

There are three ways to connect an MCP client to a remote server, depending on the client.

#### Config File (Claude Desktop, Cursor, And Others)

MCP client config files only support stdio servers. There is no way to put an HTTP URL directly in `claude_desktop_config.json`. `mcp-remote` acts as a local stdio-to-HTTP bridge. FusionAuth also does not support Dynamic Client Registration (DCR), so native clients that try to auto-register will fail. The `--static-oauth-client-info` flag bypasses DCR by providing the pre-registered client Id directly.

Update your config to point at the public URL and remove `--allow-http`:

```json
{
  "mcpServers": {
    "name-server": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://mcp.example.com/mcp",
        "3334",
        "--static-oauth-client-info",
        "{\"client_id\":\"YOUR_CLIENT_ID_HERE\"}"
      ]
    }
  }
}
```

#### CLI Tools

Some CLI tools, such as Claude Code, let you add MCP servers using a JSON config directly from the command line rather than editing a config file manually. The setup is otherwise the same as for config-file clients: `mcp-remote` handles the HTTP transport and OAuth flow, and you still need a pre-registered client Id since FusionAuth does not support DCR.

Run the setup script to register a client, then add the server using the JSON config it outputs. For example, in Claude Code:

```bash
claude mcp add-json name-server '{"command":"npx","args":["mcp-remote","https://mcp.example.com/mcp","3334","--static-oauth-client-info","{\"client_id\":\"YOUR_CLIENT_ID_HERE\"}"]}'
```

Replace `YOUR_CLIENT_ID_HERE` with the client Id from the setup script output.

#### Connector UI (Claude Desktop And Claude.ai Web)

Updated versions of Claude Desktop and claude.ai in the browser support a connector UI that connects natively over HTTP without `mcp-remote`. Navigate to <Breadcrumb>Settings -> Connectors</Breadcrumb>, add a custom connector with the URL `https://mcp.example.com/mcp`, and enter the client Id and secret from the setup script in the <InlineField>OAuth Client Id</InlineField> and <InlineField>OAuth Client Secret</InlineField> fields.

The connector UI still requires a pre-registered FusionAuth application. Use the `--connector-ui` flag when running the setup script (available in the [`protected-remote-mcp`](https://github.com/fusionauth/fusionauth-protected-mcp-server/tree/main/protected-remote-mcp) folder) to register one with the correct redirect URL and output the client secret:

```bash
python3 setup_clients.py --mcp-server-url https://mcp.example.com --connector-ui
```

The script outputs the URL, client Id, and client secret to enter in the connector UI fields.

## Troubleshooting

If you run into an issue while setting up or testing your OAuth-protected MCP server, check the following common problems and solutions.

### The "Incompatible Auth Server" Or "Does Not Support Dynamic Client Registration" Error

Your MCP client is attempting to auto-register itself as an OAuth client using Dynamic Client Registration (DCR). FusionAuth does not support DCR.

The fix depends on which client you are using:

* **Config file clients:** Use `mcp-remote` with `--static-oauth-client-info` as shown in this tutorial.
* **Connector UI (Claude Desktop, claude.ai web):** Enter the client Id and secret from the setup script in the connector settings.
* **CLI tools:** Use `claude mcp add-json` with the config output by the setup script. Run the setup script first to register the client and obtain the client Id.

### The "File Is Not Defined" Error

You're using Node.js 18 or earlier. Upgrade to Node.js 20 or later. Note that if you use nvm, GUI applications may not pick up the correct Node.js version; consider installing Node.js via your system package manager instead.

### The OAuth Flow Doesn't Start

### The Token Validation Fails

Check the MCP server logs:

```bash
docker compose logs mcp-server
```

Then verify that the user is registered to the MCP client application and that the token includes the required scopes.

### Your MCP Client Doesn't Show The Server

Verify that your configuration JSON syntax is valid, restart the client after configuration changes, and confirm the client Id matches what the setup script created.

### The OAuth Flow Completes, But Tools Aren't Available

Completely quit and restart your MCP client after the OAuth flow completes. The client needs to restart to pick up the newly obtained access token.

### The "Invalid redirect_uri" Error Occurs During OAuth

OAuth clients redirect users back to a specific URL after authentication. If FusionAuth doesn't recognize the redirect URL, it rejects the request. The setup script registers exact redirect URLs for port `3334` by default. If your MCP client uses a different port, pass `--port` to the setup script to register the correct port, or add the missing URL manually by navigating to <Breadcrumb>Applications -> [Your Client] -> OAuth -> Authorized redirect URLs</Breadcrumb> in the FusionAuth admin UI.
