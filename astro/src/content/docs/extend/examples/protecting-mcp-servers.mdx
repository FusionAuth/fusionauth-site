Building a production Model Context Protocol (MCP) server requires you to consider not just the tools and services you expose, but also how to control and track which clients can access them. With FusionAuth, you can add OAuth protection to your MCP server, letting you authenticate clients and enforce per-tool scopes, while maintaining full control with a self-hosted authorization server.

Starting with a simple, unprotected MCP server and adding OAuth protection step by step, this guide demonstrates how to:

* Examine a basic MCP server with a simple `get_name` tool.
* Add FusionAuth OAuth configuration and token validation.
* Register an MCP client, such as Claude Desktop and test the protected tool.

The following tutorial builds a working example of an OAuth-protected MCP server that you can extend with additional tools and security policies.

## Prerequisites

To follow this guide, you need the following installed:

* Docker Desktop or another Docker environment (ensure it's running).
* An MCP client such as Claude Desktop or Cursor.
* Python `3.12` or later (and `pip`).
* Node.js version `20.18.1` or later (ensure it's available in your path).
* FusionAuth `1.67.0` or later. Eearlier versions do not support the `resource` parameter.
* An Essentials or Enterprise license. This is needed for custom OAuth scopes. If you have a paid plan, find your license key at [account.fusionauth.io](https://account.fusionauth.io/account/plan/). Otherwise, [contact FusionAuth](/contact) to purchase one.

## Adding OAuth Protection

Now, let's add authentication piece by piece to protect the MCP server, starting with configuration and ending with a fully protected MCP server which runs on localhost.

### OAuth Protection For MCP Servers

To protect an MCP server, you need to add:

1. **Token Validation:** When a client sends a request, you need to verify that its access token is valid.
1. **Scope Enforcement:** MCP clients require specific scopes (permissions) to execute tools.
1. **User Context:** Once a request has been authenticated, you use the token to identify which user made the request.

This guide uses the [JWT validation API](/docs/apis/jwt#validate-a-jwt) for token validation. This API verifies the token's signature and expiration using FusionAuth's public keys and returns the token claims, including scopes, if the token is valid.

Following these steps will modify the files in `unprotected-local-mcp` but you can see the final versions in the `protected-local-mcp` directory.

## Registering MCP Clients As OAuth Clients

The first step is to register anything that is going to be requesting a token as an OAuth client. Before MCP clients like Claude Desktop or Cursor can connect to your protected MCP server, you need to register them in FusionAuth.

<Aside type="tip">
Right now all MCP client registration needs to take place out-of-band and before the MCP client connects to FusionAuth. For example, you can do this registration on-demand using the API in a developer portal. There are a couple of open issues for supporting [Dynamic Client Registration](https://github.com/fusionAuth/fusionauth-issues/issues/1125) and [Client ID Metadata Documents](https://github.com/FusionAuth/fusionauth-issues/issues/3230).
</Aside>

### The Registration Process

Each MCP client needs its own dedicated OAuth application in FusionAuth. This means if you want to support both Claude Desktop and Cursor, you register two separate applications. When an MCP client initiates the OAuth grant, it looks like this:

1. The MCP client redirects the user to FusionAuth to log in.
1. The user authenticates.
1. The user approves the requested scopes.
1. FusionAuth redirects the user back to the MCP client with an authorization code.
1. The MCP client exchanges the code for an access token.
1. The MCP client passes the token to the MCP server with each request.

For the OAuth grant to work, FusionAuth needs to know:

* Which redirect URLs are allowed for each MCP client.
* Whether PKCE is required (many MCP clients use PKCE for security).
* Which OAuth grants are enabled.
* The resource URL, which is the MCP server.

### What The Setup Script Does

The `setup_clients.py` script automates this registration process using the FusionAuth python client library. If you were integrating pre-registration into a developer portal, you'd build similar functionality using the [python client library](/docs/sdks/python) or one of [the other client libraries](/docs/sdks/). It is located in the `setup` directory. Let's look at the key parts.

The script prompts you for a name for the client, then creates an OAuth application in FusionAuth. Run it once for each MCP client you want to register. It registers exact redirect URLs for a fixed callback port (default `3334`):

<RemoteCode url={frontmatter.codeRoot + "/protected-local-mcp/setup/setup_clients.py"} lang="python" tags="create-client-application"/>

This includes the following key configuration values:

* `authorizedURLValidationPolicy: "ExactMatch"` requires the redirect URL to match one of the registered URLs exactly. This is more secure than wildcard matching. Pass `--port` to the setup script if port `3334` is already in use on your machine.
* ``authorizedResourceUris` is the list of MCP servers allowed.
* `proofKeyForCodeExchangePolicy: "Required"` enforces PKCE for security.
* `clientAuthenticationPolicy: "NotRequiredWhenUsingPKCE"` means that no client secret is needed when PKCE is used.
* `scopeHandlingPolicy: "Strict"` only allows scopes that are explicitly registered on the application.

The script returns the generated client Id and a full MCP configuration block to paste into common client configuration.

## Configuring Your MCP Client

Now let's configure the MCP client to connect to the OAuth-protected MCP server. This section covers the config file approach used by many MCP clients. CLI tools with native HTTP transport support are covered in the [Deploying A Remote MCP Server](#deploying-a-remote-mcp-server) section.

### Configuration Format

MCP clients which use configuration files to enable MCP servers often use the following configuration structure:

```json
{
  "mcpServers": {
    "name-server": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "http://localhost:8000/mcp",
        "3334",
        "--allow-http",
        "--static-oauth-client-info",
        "{\"client_id\":\"YOUR_CLIENT_ID_HERE\",\"scope\":\"openid profile get_name\"}"
      ]
    }
  }
}
```

Replace `YOUR_CLIENT_ID_HERE` with the client Id output by the registration script for the client you just set up.

This file uses the `npx` to run the `mcp-remote` npm package, because this MCP client doesn't support native HTTP connections to MCP servers. Instead they expect to communicate over standard input/output. `mcp-remote`, or a solution like it, is required to connect to any HTTP-based MCP server, regardless of whether auth is involved.

The `args` parameters do the following:

* `--allow-http` allows non-HTTPS connections for local development.
* `--static-oauth-client-info` provides the preregistered client Id from the setup script. By passing a pre-registered client Id, `mcp-remote` bypasses DCR entirely.

## How It Works

Now that you've built and tested the OAuth-protected MCP server, let's review the key concepts.

### The OAuth Flow

When an MCP client connects to the MCP server, here's what happens:

1. **Discovery:** The MCP client discovers OAuth endpoints from FusionAuth's OpenID Connect configuration.
1. **Authorization:** The MCP client redirects the user's browser to FusionAuth's login page.
1. **Authentication:** The user logs in and is redirected back to the MCP client.
1. **Token exchange:** FusionAuth returns an authorization code, which the MCP client exchanges for an access token.
1. **API calls:** The MCP client includes the access token in requests to the MCP server.
1. **Validation:** The MCP server validates tokens using FusionAuth's JWT validation endpoint before executing tools.

<Aside type="caution">
The MCP server should never forward the access token provided to downstream services. This is prohibited by the [token passthrough](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices#token-passthrough) section of the MCP security best practices. How the MCP server properly accesses such resources is beyond the scope of this document.
</Aside>

### Token Validation Approach

The MCP server validates tokens using the FusionAuth Python client's `validate_jwt` method, which calls FusionAuth's `/api/jwt/validate` endpoint. This endpoint:

* Verifies the JWT signature using FusionAuth's public keys.
* Confirms the token has not expired.
* Returns the token claims, including the `scope` field, if the token is valid.
* Does not require client credentials.

<Aside type="note">
If you try to use FusionAuth's introspect endpoint here, it will return a 401. Introspection requires that the credentials used to call the endpoint belong to the same application that issued the token. Because the MCP server is a separate FusionAuth application from each MCP client, it cannot introspect the client's tokens. See [FusionAuth issue #3010](https://github.com/FusionAuth/fusionauth-issues/issues/3010) for details. The `/api/jwt/validate` endpoint does not have this restriction.
</Aside>

### Scope Enforcement

The setup script creates the `get_name` scope on each client application it creates.

The `required_scopes=["get_name"]` parameter in the token validation ensures that:

* Only tokens containing the `get_name` scope can execute tools.
* FastMCP automatically validates scopes before tool execution.

Users must be registered to an application with this scope to obtain a token containing it.

## Production Considerations

This example demonstrates the core patterns for OAuth-protected MCP servers. For production deployments, you'll want some or all of:

* **HTTPS:** Use HTTPS for all endpoints, remove the `--allow-http` flag, and configure proper TLS certificates.
* **Token caching:** Cache token validation responses to avoid calling FusionAuth's JWT validation endpoint on every request. Set the cache TTL to match token expiration.
* **Refresh tokens:** Configure your FusionAuth application to issue refresh tokens for long-lived sessions by enabling the refresh grant and requesting the `offline_access` scope. `mcp-remote` stores tokens in `~/.mcp-auth` and handles token refresh automatically. Support for the refresh grant varies across other MCP clients, so check your client's documentation.
* **Multiple scopes:** As you add more tools, create specific scopes for each set of tools and configure the token verifier to require the appropriate scopes.
* **Monitoring:** Log token validations and tool calls in the MCP server. In FusionAuth, use the [login records API](/docs/apis/login#search-login-records) to track which users have authenticated against which MCP clients and when.
* **Rate limiting:** Implement per-user rate limits in the MCP server and return HTTP `429` responses when limits are exceeded.
* **Preregistering clients at scale:** If you need to onboard many MCP clients, automate registration by modifying the setup script to create many applications.

## Deploying A Remote MCP Server

For a production deployment, you would host the MCP server on a remote machine and connect to it over HTTPS. The MCP server code requires no changes. The only differences are infrastructure, configuration, and how MCP clients connect.

### Infrastructure Requirements

For a remote deployment you need:

* A server hosting your MCP server, accessible over the internet.
* A reverse proxy such as [Caddy](https://caddyserver.com/) (which handles TLS certificates automatically) or TLS configured directly on your load balancer to serve HTTPS on the MCP server endpoint.
* A publicly accessible FusionAuth instance, since the OAuth browser redirect requires the user's browser to reach FusionAuth.

### Server Configuration

The [`protected-remote-mcp`](https://github.com/fusionauth/fusionauth-example-protected-mcp-server/tree/main/protected-remote-mcp) folder of the example repository contains a `docker-compose.yml` that runs only the MCP server, with no FusionAuth or database. Copy it to your server and set the following environment variables when starting:

* `MCP_SERVER_URL`. Set this to your server's public HTTPS URL (e.g. `https://mcp.example.com`). FastMCP uses this to construct the OAuth metadata that MCP clients discover.
* `FUSIONAUTH_URL`. Set this to your FusionAuth instance's public URL. The MCP server uses this to validate tokens internally.
* `FUSIONAUTH_EXTERNAL_URL`. Set this to the same value. This is the URL that MCP clients use in their browser to complete the OAuth flow. In containerized deployments where FusionAuth is reachable internally at a different address, the two can differ.

```console
MCP_SERVER_URL=https://mcp.example.com \
FUSIONAUTH_URL=https://auth.example.com \
FUSIONAUTH_EXTERNAL_URL=https://auth.example.com \
docker compose up -d
```

### Registering Remote MCP Clients

Run the setup script from the `protected-remote-mcp/setup` directory, passing `--fusionauth-url`, `--api-key`, and `--mcp-server-url` to point it at your remote FusionAuth instance and MCP server:

```console
python setup_clients.py \
  --fusionauth-url https://auth.example.com \
  --api-key YOUR_API_KEY \
  --mcp-server-url https://mcp.example.com
```

If your FusionAuth instance has multiple tenants, also pass `--tenant-id` to target the correct one:

```console
python setup_clients.py \
  --fusionauth-url https://auth.example.com \
  --api-key YOUR_API_KEY \
  --mcp-server-url https://mcp.example.com \
  --tenant-id YOUR_TENANT_ID
```

### Connecting MCP Clients

There are a few ways to connect an MCP client to a remote server, depending on the client.

#### Config File 

This is used for clients like Claude Desktop and Cursor.

MCP client config files only support stdio servers. There is no way to put an HTTP URL directly in `claude_desktop_config.json`. `mcp-remote` acts as a local stdio-to-HTTP bridge for remote servers, the same as it did for the server running over HTTP on localhost.

The `--static-oauth-client-info` flag bypasses just-in-time registration of MCP clients by providing the pre-registered client Id directly.

Update your config to point at the public URL and remove `--allow-http` and the `port`. Your config will look something like this:

```json
{
  "mcpServers": {
    "name-server": {
      "command": "npx",
      "args": [
        "mcp-remote",
        "https://mcp.example.com/mcp",
        "--static-oauth-client-info",
        "{\"client_id\":\"YOUR_CLIENT_ID_HERE\",\"scope\":\"openid profile get_name\"}"
      ]
    }
  }
}
```

#### CLI Tools

Some CLI tools, such as Claude Code, let you add MCP servers using a JSON config directly from the command line rather than editing a config file manually. The setup is otherwise the same as for config-file clients: `mcp-remote` handles the HTTP transport and OAuth flow, and you still need a pre-registered client Id.

Run the setup python script to register a client, then add the server using the JSON config it outputs. For example, in Claude Code:

```console
claude mcp add-json name-server '{"command":"npx","args":["mcp-remote","https://mcp.example.com/mcp","3334","--static-oauth-client-info","{\"client_id\":\"YOUR_CLIENT_ID_HERE\",\"scope\":\"openid profile get_name\"}"]}'
```

This will vary by the tool, but in general you'll need to provide the client Id from the setup script output. 

#### Agents And Other MCP Clients

Consult the documentation for your preferred agentic frameworks or other MCP clients.

## Troubleshooting

If you run into an issue while setting up or testing your OAuth-protected MCP server, check the following common problems and solutions.

### The "Incompatible Auth Server" Or "Does Not Support Dynamic Client Registration" Error

Your MCP client is attempting to auto-register itself as an OAuth client using Dynamic Client Registration (DCR). FusionAuth does not support this.

Run the setup script first to register the client and obtain the client Id. Then, depending on which client you are using, make sure to configure the client Id:

* **Config file clients:** Use `mcp-remote` with `--static-oauth-client-info`.
* **CLI tools:** Use `claude mcp add-json` with `--static-oauth-client-info`.

### The "File Is Not Defined" Error

You're using Node.js 18 or earlier. Upgrade to Node.js 20 or later. Note that if you use nvm, GUI applications may not pick up the correct Node.js version; consider installing Node.js via your system package manager instead.

### The OAuth Flow Doesn't Start

Check that the Docker services are running:

```console
docker compose ps
```

Verify that the MCP server responds:

```console
curl http://localhost:8000/.well-known/oauth-protected-resource/mcp
```

### The Token Validation Fails

Check the MCP server logs:

```console
docker compose logs mcp-server
```

Then verify that the user is registered to the MCP client application and that the token includes the required scopes.

### Your MCP Client Doesn't Show The Server

Verify that your configuration JSON syntax is valid, restart the client after configuration changes, and confirm the client Id matches what the setup script created.

### The OAuth Flow Completes, But Tools Aren't Available

Completely quit and restart your MCP client after the OAuth flow completes. The client needs to restart to pick up the newly obtained access token.

If you are using `mcp-remote`, delete the `~/.mcp-auth/` directory to remove cached tokens.

### The "Invalid redirect_uri" Error Occurs During OAuth

OAuth clients redirect users back to a specific URL after authentication. If FusionAuth doesn't recognize the redirect URL, it rejects the request. The setup script registers exact redirect URLs for port `3334` by default. If your MCP client uses a different port, pass `--port` to the setup script to register the correct port, or add the missing URL manually by navigating to <Breadcrumb>Applications -> [Your Client] -> OAuth -> Authorized redirect URLs</Breadcrumb> in the FusionAuth admin UI.
