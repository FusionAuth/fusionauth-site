This guide covers operational security hardening for FusionAuth in production. It is organized into three parts:

- **Universal Hardening** — applies to every FusionAuth deployment regardless of hosting model.
- **Self-Hosted Infrastructure** — applies only when you operate the servers, database, and search engine yourself.
- **Deployment-Specific Guidance** — additional hardening for FusionAuth Cloud, Docker, Kubernetes, bare metal, multi-node clusters, and air-gapped environments.

Within each part, sections are independent — implement what applies to your deployment. The deployment-specific sections build on top of the universal and self-hosted sections, so start with those before moving to your deployment type.

## Universal Hardening

The following applies to all FusionAuth deployments — self-hosted and FusionAuth Cloud.

### Admin Account Security

On a fresh FusionAuth install, the initial setup creates a single User with the `admin` role. No MFA is configured and no access restrictions are applied.

**Enable MFA for admin accounts.** Navigate to <Breadcrumb>Tenants -> Your Tenant -> Multi-Factor</Breadcrumb> and enable at least one MFA method. Basic MFA (TOTP authenticator app) is available on all plans including Community. Advanced MFA (email, SMS, and voice) requires the Starter plan or above. Admin Users can then enroll via their account settings in the Admin UI. See [Multi-Factor Authentication](/docs/lifecycle/authenticate-users/multi-factor-authentication) for configuration details.

<Aside type="note">
Application-level MFA policy overrides (requiring MFA only for the FusionAuth Admin Application while leaving it optional for other Applications) require the Enterprise plan. On other plans, MFA policy is Tenant-wide.
</Aside>

**Use least-privilege admin roles.** FusionAuth provides [38 built-in roles](/docs/get-started/core-concepts/roles) for the Admin Application, including `admin` (full access), `user_manager`, `user_support_manager` (tier-1 support), `audit_log_viewer`, `api_key_manager`, and others. Assign Users the minimum role required for their function rather than granting `admin` to everyone. Custom admin roles are not supported — use the built-in set.

### Password Policy

FusionAuth's default password policy on a fresh install requires only a minimum length of 8 characters. No complexity requirements are enabled.

Navigate to <Breadcrumb>Tenants -> Your Tenant -> Password</Breadcrumb> to configure:

| Setting | Default | Recommendation |
|---------|---------|----------------|
| Minimum length | 8 | 10–12 for User accounts, 16+ for admin and service accounts |
| Require mixed case | Off | Enable |
| Require number | Off | Enable |
| Require non-alphanumeric | Off | Consider enabling |
| Password history | Off | Enable with 3–5 previous passwords |
| Breached password detection | Off | Enable if licensed (Starter plan and above) |

**Breached password detection** checks passwords against a dataset of known compromised credentials in real time. It is available on any paid plan (Starter and above) and requires outbound network connectivity to FusionAuth servers. When enabled, it blocks compromised passwords at registration and password change. For login, you can configure the behavior to record the event, notify the User via email, or force a mandatory password reset before granting access. See [Breached Password Detection](/docs/operate/secure/breached-password-detection) for details.

### Session and Token Management

#### Access Token Lifetime

Access tokens are short-lived credentials issued after authentication. Configure the lifetime at the Tenant level under <Breadcrumb>Tenants -> Your Tenant -> JWT</Breadcrumb>. Applications can override the Tenant default under <Breadcrumb>Applications -> Your App -> JWT</Breadcrumb> by enabling **Override tenant JWT configuration**.

For security-sensitive deployments, keep the access token lifetime short — 5 to 15 minutes. Shorter lifetimes limit the window of exposure if a token is compromised.

#### Refresh Token Configuration

Refresh tokens control how long a User stays authenticated without re-entering credentials. Key settings under <Breadcrumb>Tenants -> Your Tenant -> JWT -> Refresh token</Breadcrumb>:

| Setting | Default | Recommendation |
|---------|---------|----------------|
| Usage policy | `Reusable` | `OneTimeUse` — each use issues a new token and invalidates the old one |
| Expiration policy | `Fixed` | `SlidingWindowWithMaximumLifetime` — active sessions persist with an absolute upper bound |
| Lifetime | Set during Tenant creation (required, no preset default) | Reduce based on your risk tolerance |

`OneTimeUse` refresh tokens enable **token rotation** — each use of a refresh token issues a new token and invalidates the previous one. This limits the window in which a stolen refresh token can be used. See [Tokens](/docs/lifecycle/authenticate-users/oauth/tokens) for details.

#### SSO Session Timeout

FusionAuth creates an SSO session when the `rememberDevice` parameter is set to `true` during login. On the hosted login page, this is controlled by the "Keep me signed in" checkbox — though the Theme can hardcode this value to always create SSO sessions without user interaction. SSO sessions allow authentication to other Applications in the same Tenant without re-entering credentials. Configure the session timeout at <Breadcrumb>Tenants -> Your Tenant -> OAuth -> Session timeout</Breadcrumb>. The default is 3600 seconds (1 hour). See [Single Sign-On](/docs/lifecycle/authenticate-users/single-sign-on) for details.

#### Invalidating Sessions

FusionAuth does not provide built-in concurrent session limiting. To enforce single-session access, listen for the `user.login.success` [webhook](/docs/extend/events-and-webhooks/) event and revoke existing refresh tokens for the User via the API.

To terminate all active sessions for a specific User, revoke their refresh tokens: navigate to <Breadcrumb>Users -> [User]</Breadcrumb> and select **Delete All Refresh Tokens**, or call `DELETE /api/jwt/refresh?userId={userId}`.

### Login Form Security

FusionAuth's hosted login pages provide several protections against account enumeration by default:

- **Login failure messages.** The default error message is the same whether the username does not exist or the password is incorrect.
- **Forgot password flow.** The response is identical regardless of whether the submitted email address exists. If the email does not match a User, no email is sent, but the response is the same.

Two account states produce distinct responses on the hosted login page that confirm an account exists:

- **Locked accounts.** The default Theme displays a distinct locked account message. To normalize this, edit the locked account message in your Theme's message properties to use the same text as the generic login failure message. See the [FusionAuth localization repository](https://github.com/FusionAuth/fusionauth-localization) for the default message keys.
- **Email verification required.** FusionAuth displays a verification prompt instead of a login failure. You can customize the page in your Theme, but you cannot suppress the flow without disabling email verification.

Two additional states produce distinct responses only after valid credentials have been submitted. These confirm that the credentials are correct, not merely that an account exists:

- **Multi-factor authentication.** The MFA challenge screen appears after successful first-factor authentication. This flow cannot be hidden without disabling MFA.
- **Password change required.** A redirect to the password change form occurs after successful authentication. This flow cannot be hidden without disabling forced password changes.

These post-authentication flows are standard in identity providers. Hiding them is not recommended, as doing so would degrade core security functionality.

### Multi-Tenant Enumeration Prevention

In a multi-tenant FusionAuth deployment, each Tenant has its own Applications with distinct OAuth `client_id` values. A common concern during security audits is **tenant enumeration** — an attacker probing the authorization endpoint to discover which Tenants or Applications exist.

When `/oauth2/authorize` receives a valid `client_id`, FusionAuth renders a login page. When it receives an invalid `client_id`, FusionAuth renders an error page. This differential response is required by the OAuth 2.0 specification ([RFC 6749 §4.1.2.1](https://datatracker.ietf.org/doc/html/rfc6749#section-4.1.2.1)) — the authorization server must not redirect when the client identifier is invalid.

**Server-side tenant resolution.** Your application should resolve the Tenant and `client_id` on the server side before redirecting the User to FusionAuth. Users should never construct or see the `/oauth2/authorize` URL directly. When your application cannot resolve a Tenant identifier, return the same response as a valid Tenant rather than forwarding the unresolved request to FusionAuth.

If you are using the [Login API](/docs/apis/login) directly, your application controls the full response and can normalize error messages and timing. If you are using [hosted login pages](/docs/get-started/core-concepts/integration-points#hosted-login-pages), Tenant resolution must happen in your application before the redirect to `/oauth2/authorize`.

**Subdomain-per-tenant with proxy routing.** Assign each Tenant a subdomain (for example, `tenant-a.auth.example.com`). Your reverse proxy maps the subdomain to the correct `tenantId` by appending it as a query parameter on the authorize URL or by setting the `X-FusionAuth-TenantId` header on API requests. Use the query parameter for hosted login page flows and the header for direct API calls.

**Error page theming.** The default OAuth error template displays a JSON block containing the error type and the literal `client_id` value. Navigate to <Breadcrumb>Customizations -> Themes</Breadcrumb>, edit your Theme, and select the **OAuth error** template. Remove the `${oauthJSONError}` block and replace it with a generic message. Error detail remains available in FusionAuth's server-side event log for debugging.

<Aside type="note">
When FusionAuth receives an invalid `client_id`, it cannot resolve an Application, so the Application-level Theme cannot be determined. FusionAuth falls back to the Tenant-level Theme (if a valid `tenantId` was provided) or the default Tenant's Theme. Apply your customization to the Theme used by your default Tenant in addition to any Application-specific Themes.
</Aside>

Even with a customized error page, the response is structurally different from a login page. Server-side resolution is the primary defense; theming is defense in depth.

### Authorization Code Security (PKCE)

The OAuth 2.0 Authorization Code grant transmits authorization codes in URL query strings — they appear in browser history, server access logs, proxy logs, and HTTP `Referer` headers. [Proof Key for Code Exchange (PKCE)](https://datatracker.ietf.org/doc/html/rfc7636) binds each authorization code to the client session that initiated it, preventing an intercepted code from being exchanged by a different party.

Configure PKCE per Application at <Breadcrumb>Applications -> Your App -> OAuth -> PKCE</Breadcrumb>. Three options are available:

| Setting | Behavior |
|---------|----------|
| `Required` | The authorization request must include `code_challenge` and `code_challenge_method`. The token exchange must include `code_verifier`. Requests without PKCE parameters are rejected. |
| `Not required when using client authentication` | PKCE is required for public clients (those that do not send a `client_secret`). Confidential clients that authenticate at the token endpoint are exempt. |
| `Not required` | PKCE is accepted but not enforced. This is the default. |

For production deployments, set PKCE to `Required` or `Not required when using client authentication`. This is consistent with [OAuth 2.1](https://fusionauth.io/articles/oauth/differences-between-oauth-2-oauth-2-1), which mandates PKCE for all Authorization Code grants.

Applications using `http://` redirect URIs are especially exposed — authorization codes travel in cleartext and are observable by any network intermediary. PKCE is the primary defense against code interception in these configurations.

### CORS Configuration

FusionAuth's CORS filter is **disabled by default**. When disabled, FusionAuth does not send CORS headers — cross-origin API calls from browser Applications will fail.

If your Application makes direct API calls to FusionAuth from a browser (for example, a single-page application calling `/oauth2/token`), enable and configure the CORS filter at <Breadcrumb>Settings -> System -> CORS</Breadcrumb>.

When enabled, **Allowed methods** defaults to `GET` and `OPTIONS` only. You must add `POST` for the token endpoint and any other methods your Application uses.

<Aside type="caution">
Do not use `*` as an allowed origin when **Allow credentials** is enabled. The CORS specification forbids this combination and browsers will reject the response. List each Application origin explicitly.
</Aside>

If you encounter unexpected CORS 403 errors, enable **Debug mode** under CORS settings to generate event log entries for rejected requests. See the [CORS Reference](/docs/operate/secure/cors) for full configuration details.

Server-side OAuth flows (authorization code with a backend) do not require CORS configuration because the browser never makes direct cross-origin requests to FusionAuth.

### Cryptographic Key Management

FusionAuth manages cryptographic keys through [Key Master](/docs/operate/secure/key-master) at <Breadcrumb>Settings -> Key Master</Breadcrumb>. Key Master handles RSA, ECDSA, HMAC, and EdDSA keys used for JWT signing, SAML assertions, and webhook signatures.

<Aside type="version">
EdDSA (Ed25519) key support is available since version `1.62.0`.
</Aside>

#### Rotating Signing Keys

To rotate a JWT signing key with no downtime:

1. Create the new asymmetric key in Key Master with the same algorithm as the current key. The key is available in `/.well-known/jwks.json` alongside the existing key. (HMAC keys are not published to the JWKS endpoint.)
3. Update the Tenant or Application JWT configuration to sign new tokens with the new key.
4. Wait until all tokens signed by the old key have expired (at minimum, the access token TTL).
5. Remove the old key assignment and delete it from Key Master.

Resource servers that validate tokens by fetching the JWKS endpoint will see both keys during the overlap window and can verify tokens signed by either key.

<Aside type="note">
FusionAuth prevents deletion of a key that is actively assigned to a Tenant or Application. You must reassign the signing configuration before deleting the old key.
</Aside>

Establish a key rotation schedule as part of your operational procedures. FusionAuth does not enforce automatic key expiration or rotation. See [Key Rotation](/docs/operate/secure/key-rotation) for additional guidance.

### API Key Management

FusionAuth API Keys should follow the principle of least privilege. Navigate to <Breadcrumb>Settings -> API Keys</Breadcrumb> to create or edit keys.

**Tenant-scoped keys.** In multi-tenant environments, use the **Tenant** dropdown in the key editor to scope each API Key to a specific Tenant. A Tenant-scoped key can only access resources within its assigned Tenant.

**Endpoint restrictions.** The **Endpoints** section in the key editor lists every API endpoint with toggles for each HTTP method. Restrict each key to only the endpoints it needs. An API Key with no endpoint restrictions is a super-key with full API access.

**Key rotation.** FusionAuth does not enforce automatic API Key expiration. To rotate a key with no downtime, create the new key, update all consumers, verify the old key is no longer in use, then delete it. Running two keys in parallel during the transition avoids service interruption.

### Monitoring and Audit Logging

#### Webhook Configuration

FusionAuth delivers security events via [webhooks](/docs/extend/events-and-webhooks/). Setting up webhooks requires two steps:

1. **Create the webhook.** Navigate to <Breadcrumb>Settings -> Webhooks</Breadcrumb> and add a new webhook with the target URL and signing secret.
2. **Enable events at the Tenant level.** Navigate to <Breadcrumb>Tenants -> Your Tenant -> Webhooks</Breadcrumb> and enable each event type you want to receive.

<Aside type="caution">
Both steps are required. If you create a webhook but do not enable the event types at the Tenant level, the webhook will never fire.
</Aside>

#### Key Events to Monitor

- `user.login.failed` — repeated failures from the same IP or targeting the same account may indicate credential stuffing or brute-force attacks.
- `user.login.success` — monitor for logins from unexpected locations or at unusual times.
- `user.login.suspicious` — fires when Advanced Threat Detection identifies a login as a potential threat. Requires the Enterprise plan.
- `user.password.breach` — fires when a breached password is detected during login. Requires any paid plan (Starter and above).

#### Webhook Security

- Use HTTPS endpoints for all webhook URLs.
- Configure a webhook signing key so your receiver can verify that events originated from FusionAuth.
- Validate the signature on every received event before processing.

See [Securing Webhooks](/docs/extend/events-and-webhooks/securing) for details.

#### Audit Log

The FusionAuth audit log records administrative actions such as User creation, Application configuration changes, and key modifications. Review this log regularly and forward it to a centralized logging system.

<Aside type="note">
In multi-tenant deployments, audit log records are not isolated by Tenant. All administrative actions across all Tenants appear in the same audit log. If your compliance requirements mandate Tenant-isolated audit trails, consider running separate FusionAuth instances per Tenant.
</Aside>

## Self-Hosted Infrastructure

The following applies only when you operate the servers, database, and search engine yourself. If you are using [FusionAuth Cloud](/docs/get-started/run-in-the-cloud/cloud), these concerns are managed for you — skip to [Deployment-Specific Guidance](#deployment-specific-guidance).

### Reverse Proxy

Running FusionAuth behind a reverse proxy is the recommended production architecture. While FusionAuth can terminate TLS directly (via `fusionauth-app.https.enabled`, available since version 1.43.0 — see the [configuration reference](/docs/reference/configuration)), most production deployments handle TLS termination at the proxy layer. See the [proxy setup guide](/docs/operate/deploy/proxy-setup) for base configuration.

Your reverse proxy should enforce TLS 1.2 or higher for all client-facing traffic and redirect HTTP requests to HTTPS. FusionAuth communicates with the proxy over HTTP on a private network — ensure this internal traffic is not exposed to public networks.

**Restrict Admin UI access.** FusionAuth's IP access control lists apply to OAuth, account, and registration endpoints — not to the Admin UI directly. To restrict Admin UI access by IP, configure your reverse proxy to block requests to `/admin/` from unauthorized source IPs.

### Trusted Proxy Configuration

FusionAuth uses the `X-Forwarded-For` header to determine client IP addresses for audit logging, rate limiting, and IP ACLs. By default, the trust policy is set to `All`, which means FusionAuth trusts every proxy in the forwarding chain. An attacker can exploit this by injecting arbitrary values into the `X-Forwarded-For` header to bypass IP-based access control lists or poison audit logs.

Navigate to <Breadcrumb>Settings -> System -> Networking</Breadcrumb> and set the trust policy to `Only Configured`. Add the IP addresses of your reverse proxy servers to the trusted proxies list. See [Networking Configuration](/docs/operate/secure/networking) for details.

<Aside type="version">
The trusted proxy configuration is available since version `1.49.0`.
</Aside>

<Aside type="caution">
If you have an IP-based access control list configured on an Application, saving an incorrect set of trusted proxies can make OAuth endpoints inaccessible. Verify your proxy IP addresses and ensure you have a direct-access fallback before changing the trust policy.
</Aside>

### Pre-Authentication Rate Limiting

FusionAuth's [Advanced Threat Detection](/docs/operate/secure/advanced-threat-detection) provides rate limiting for identity-scoped events: failed logins, forgot password requests, email verification, passwordless login, registration verification, two-factor code delivery, and phone verification. These rate limits apply per email address or phone number.

Requests that do not involve an identity — such as probing `/oauth2/authorize` with invalid `client_id` values to enumerate Tenants — are not covered by these rate limits. Configure per-IP rate limiting in your reverse proxy or WAF for the `/oauth2/` endpoints to protect against this type of brute-force probing.

<Aside type="caution">
Set the burst value high enough to accommodate legitimate traffic spikes. In enterprise SSO deployments, a single User opening multiple Applications simultaneously generates a burst of concurrent requests to `/oauth2/authorize`. A burst value that is too low causes intermittent 503 errors on login initiation.
</Aside>

### Database Connection Security

FusionAuth connects to your database (PostgreSQL or MySQL) over an unencrypted connection by default. Enable TLS by appending SSL parameters to the JDBC URL in `fusionauth.properties`:

```properties
database.url=jdbc:postgresql://hostname:5432/fusionauth?ssl=true&sslmode=verify-full
```

Standard PostgreSQL SSL modes apply: `require` (encrypt, no certificate verification), `verify-ca` (verify the server certificate CA), and `verify-full` (verify CA and hostname). Use `verify-full` when connecting over an untrusted network. See the [configuration reference](/docs/reference/configuration) for the full set of database properties.

<Aside type="note">
If you deploy FusionAuth via Helm, the chart provides `database.tls` and `database.tlsMode` values as a convenience wrapper. For non-Helm deployments, configure TLS directly in the JDBC URL as shown above.
</Aside>

For MySQL deployments on managed services that require explicit TLS version selection, append `enabledTLSProtocols=TLSv1.2` to the connection URL or set `FUSIONAUTH_APP_ADDITIONAL_JAVA_ARGS="-Djdk.tls.client.protocols=TLSv1.2"`.

### Search Engine Hardening

FusionAuth's bundled search service (Elasticsearch) stores a full index of User data — email addresses, usernames, User Ids, registration data, and custom data. Unauthenticated access to this index is a critical exposure.

The bundled search binds to localhost (`_local_`) by default on ports `9020` (cluster coordination) and `9021` (HTTP). For single-node deployments, this is the correct configuration — no external ports should be opened.

For external Elasticsearch or OpenSearch clusters, HTTP basic authentication is supported (since version 1.16.0). Credentials are embedded in the URL and stored in plaintext in `fusionauth.properties` — protect this file with appropriate filesystem permissions (see [Bare Metal and Package Installs](#bare-metal-and-package-installs)):

```properties
fusionauth-app.search-servers=https://user:password@myelasticsearchcluster.com:9200
```

FusionAuth supports Elasticsearch 7.6.1–7.17.x and OpenSearch 2.x (since version 1.42.0). See the [system requirements](/docs/get-started/download-and-install/system-requirements) for current compatibility.

If the external cluster uses a custom CA certificate, import it into the Java truststore. See [Custom Certificate Authority](/docs/operate/secure/#custom-certificate-authority).

### HTTP Security Headers

FusionAuth sets `X-Frame-Options: DENY` on hosted login pages by default. Other security headers must be configured in your reverse proxy:

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

Do not add a second `X-Frame-Options` header — FusionAuth already sets this and duplicate headers cause inconsistent behavior across browsers.

<Aside type="note">
If you need to allow FusionAuth login pages to load in an iframe for a specific origin, add the URL to **Authorized request origin URLs** in the Application's OAuth configuration. The recommended approach for token renewal is to use refresh tokens rather than iframe-based silent renewal.
</Aside>

**Content Security Policy.** FusionAuth's default Themes use inline JavaScript and inline styles. A strict CSP that blocks `unsafe-inline` for `script-src` will break the hosted login pages. Audit your Theme for inline scripts before deploying a CSP. Start with `Content-Security-Policy-Report-Only` to identify violations before enforcing restrictions. See [Advanced Themes](/docs/customize/look-and-feel/advanced-themes/) for details on custom Theme development.

### Kickstart Security

[Kickstart](/docs/get-started/download-and-install/development/kickstart) is a JSON-based bootstrapping mechanism that pre-configures FusionAuth on first boot. It runs only when no API Keys, Users, or Tenants exist — it will not execute on a previously configured instance.

Kickstart files can contain sensitive data: API Keys, User passwords, client secrets, and any value you would otherwise submit via the API. Use environment variable substitution to keep secrets out of the file:

```json
{
  "variables": {
    "apiKey": "#{ENV.FUSIONAUTH_API_KEY}",
    "adminPassword": "#{ENV.FUSIONAUTH_ADMIN_PASSWORD}"
  }
}
```

After initial provisioning, remove or restrict access to the Kickstart file on disk. FusionAuth does not delete or disable the file automatically.

### Upgrades and Patching

Keep FusionAuth up to date to receive security fixes. The upgrade process:

1. **Back up the database.** Database migrations are forward-only — there is no automated rollback.
2. **Review release notes** for every version between your current version and the target. Pay attention to breaking changes in Themes, APIs, and configuration.
3. **Test in a non-production environment** before upgrading production, especially when skipping multiple versions.
4. **Stop FusionAuth, deploy the new version, and start it.** Database migrations run automatically on startup.

FusionAuth supports upgrading across multiple minor versions, though the recommended practice is to migrate one minor version at a time and test at each step. Custom Themes may break when skipping versions that introduced new FreeMarker templates. If the hosted login page returns a 500 error after upgrading, append `?bypassTheme=true` to the URL to access the default Theme and diagnose the issue.

To roll back a failed upgrade, stop FusionAuth, restore the pre-upgrade database backup, and redeploy the previous version.

**Monitoring for security releases.** FusionAuth publishes security-relevant updates on the [vulnerabilities page](/docs/operate/secure/vulnerabilities) and in the [release notes](/docs/release-notes/). Monitor these pages to stay informed of security fixes. See the [upgrade guide](/docs/operate/deploy/upgrade) for detailed procedures.

## Deployment-Specific Guidance

### FusionAuth Cloud

[FusionAuth Cloud](/docs/get-started/run-in-the-cloud/cloud) is a fully managed deployment. FusionAuth operates the database, search engine, TLS certificates, backups, and infrastructure security. Your responsibility is limited to product-level configuration.

**What FusionAuth Cloud manages for you:**
- TLS termination and certificate provisioning (including custom domains)
- Database operations and encryption
- Search service (Elasticsearch)
- DDoS protection and network firewalls
- Infrastructure patching and upgrades (you schedule the timing)
- Backups (Business and HA tiers: 3-day retention)

**What you are responsible for:**
- All items in [Universal Hardening](#universal-hardening) above
- Application and Tenant configuration
- API Key scoping and rotation
- Theme customization
- Webhook configuration and endpoint security

**Sections of this guide that do not apply to Cloud deployments** (managed by FusionAuth):

- Reverse Proxy
- Trusted Proxy Configuration
- Pre-Authentication Rate Limiting
- Database Connection Security
- Search Engine Hardening
- HTTP Security Headers
- Kickstart Security
- Upgrades and Patching

<Aside type="note">
If you place your own reverse proxy (such as Cloudflare or Nginx) in front of FusionAuth Cloud, ensure it forwards the correct client IP. Proxies that mask client IPs can undermine FusionAuth Cloud's built-in DDoS protection. See the [proxy setup guide](/docs/operate/deploy/proxy-setup) for FusionAuth Cloud proxy requirements.
</Aside>

**Subdomain-per-tenant on Cloud.** FusionAuth Cloud does not natively map custom domains to specific Tenants. To use subdomain-per-tenant routing, place your own reverse proxy in front of FusionAuth Cloud and configure Tenant routing there.

### Docker

The [Docker installation guide](/docs/get-started/download-and-install/docker) provides Compose examples that are designed for development and do not include production security configuration.

For production Docker deployments, apply all items in [Self-Hosted Infrastructure](#self-hosted-infrastructure) above, plus:

**Secrets management.** Do not hardcode database passwords, API Keys, or other secrets in your `docker-compose.yml` or `.env` files committed to version control. Use Docker Secrets, a secrets manager (such as HashiCorp Vault or AWS Secrets Manager), or environment variable injection from your orchestration platform.

**Network isolation.** The default Docker bridge network allows communication between all containers on the same bridge. Create a dedicated network for FusionAuth and its database, and do not attach unrelated services to it.

**Container user.** The official FusionAuth Docker image runs as a dedicated `fusionauth` user (non-root). If your orchestration platform enforces `runAsNonRoot`, the image is compatible by default.

**Image provenance.** Use only official FusionAuth images from Docker Hub (`fusionauth/fusionauth-app`). Pin to a specific version tag rather than using `latest` to ensure reproducible deployments.

**Kickstart in containers.** In container environments, pass the Kickstart configuration via environment variable injection (for example, a Kubernetes Secret mounted as a file) rather than baking it into the image. This avoids persisting secrets in image layers.

### Kubernetes

FusionAuth provides deployment guides for [EKS](/docs/get-started/download-and-install/kubernetes/eks), [GKE](/docs/get-started/download-and-install/kubernetes/gke), [AKS](/docs/get-started/download-and-install/kubernetes/aks), and [Helm](/docs/get-started/download-and-install/kubernetes/fusionauth-deployment). These guides cover provisioning but do not address Kubernetes security hardening in depth.

For production Kubernetes deployments, apply all items in [Self-Hosted Infrastructure](#self-hosted-infrastructure) above, plus:

**Secrets management.** Store database credentials and API Keys in Kubernetes Secrets. The FusionAuth Helm chart supports referencing a pre-existing Secret for database credentials — the Secret must contain a `password` key and optionally a `rootpassword` key. See the [Kubernetes deployment guide](/docs/get-started/download-and-install/kubernetes/fusionauth-deployment) for configuration details. For stronger isolation, use an external secrets manager integrated via the External Secrets Operator, HashiCorp Vault Agent, or your cloud provider's secrets CSI driver:
- AWS: [Secrets Manager with EKS](https://docs.aws.amazon.com/secretsmanager/latest/userguide/integrating_csi_driver.html)
- GCP: [Secret Manager with GKE](https://cloud.google.com/secret-manager/docs/using-other-products#google-kubernetes-engine)
- Azure: [Key Vault with AKS](https://learn.microsoft.com/en-us/azure/aks/csi-secrets-store-driver)

**Network policies.** Restrict traffic to FusionAuth pods so that only your application pods and the load balancer can reach port 9011. Deny ingress from all other sources. Restrict egress to the database, search service, and any required external services (such as SMTP or webhook targets):
- [Kubernetes network policies](https://kubernetes.io/docs/concepts/services-networking/network-policies/)
- [EKS network policy support](https://docs.aws.amazon.com/eks/latest/userguide/cni-network-policy.html)
- [GKE network policy](https://cloud.google.com/kubernetes-engine/docs/how-to/network-policy)
- [AKS network policies](https://learn.microsoft.com/en-us/azure/aks/use-network-policies)

**Pod security.** Apply Pod Security Standards to FusionAuth pods. At minimum, configure a security context that drops all capabilities and prevents privilege escalation:
- [Kubernetes Pod Security Standards](https://kubernetes.io/docs/concepts/security/pod-security-standards/)

**Ingress and TLS.** Terminate TLS at the Ingress controller or load balancer. Use cert-manager or your cloud provider's certificate service for automated certificate provisioning and renewal. Ensure the Ingress forwards the `X-Forwarded-For` and `X-Forwarded-Proto` headers to FusionAuth.

**Database access.** Use managed database services (RDS, Cloud SQL, Azure Database for PostgreSQL) with private networking. Do not expose the database to the public internet. Restrict the database security group or firewall to accept connections only from the FusionAuth pod CIDR or service account.

### Bare Metal and Package Installs

The [FusionAuth application installation guide](/docs/get-started/download-and-install/fusionauth-app) covers package-based installs on Linux, macOS, and Windows. For production bare metal deployments, apply all items in [Self-Hosted Infrastructure](#self-hosted-infrastructure) above, plus:

**Service user isolation.** Run FusionAuth under a dedicated service account with no login shell and minimal filesystem permissions. Do not run FusionAuth as root.

**Configuration file security.** The `fusionauth.properties` file contains database credentials and other sensitive configuration. Restrict file permissions so that only the FusionAuth service user can read it.

**Runtime mode.** Set `fusionauth-app.runtime-mode=production` in `fusionauth.properties`. Development mode enables Maintenance Mode, which exposes sensitive configuration through a web interface. See the [configuration reference](/docs/reference/configuration) for details.

**Default credentials.** FusionAuth example configurations use `fusionauth` as the default database username. Change all database credentials to strong, unique values before deploying to production. See the [configuration reference](/docs/reference/configuration) for the relevant properties.

### Multi-Node Clusters

The [cluster deployment guide](/docs/operate/deploy/cluster) covers multi-node FusionAuth installations. In addition to all single-node hardening:

**Inter-node communication.** All FusionAuth nodes must communicate over a private network. If nodes span network boundaries, use TLS for inter-node communication and install the required certificates in each node's Java truststore. See [Custom Certificate Authority](/docs/operate/secure/#custom-certificate-authority) for truststore import instructions.

**Load balancer as sole entry point.** FusionAuth nodes should not be directly accessible from the public internet. Route all traffic through a load balancer. Configure the load balancer to forward `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`, and `X-Forwarded-Server` headers.

**Consistent configuration.** All nodes must share identical configuration — credentials, plugins, runtime mode, and FusionAuth version. Inconsistent configuration produces undefined behavior.

**Upgrades.** FusionAuth supports a rolling upgrade procedure: remove a node from the load balancer, upgrade it, add it back, and repeat. However, FusionAuth does not support zero-downtime upgrades when there is a database schema change — the first upgraded node acquires a migration lock and other nodes may produce errors until they are upgraded. Plan for a brief service interruption during schema migrations. See the [upgrade guide](/docs/operate/deploy/upgrade) for the full cluster upgrade procedure.

### Air-Gapped Deployments

The [air-gapping guide](/docs/get-started/download-and-install/reference/air-gapping) covers deploying FusionAuth without outbound internet access. Air-gapped deployments require an Enterprise license with manual activation.

**Feature limitations.** Two security features require outbound connectivity to FusionAuth servers and are unavailable in air-gapped environments:
- **Breached password detection** — passwords cannot be checked against the compromised credentials dataset.
- **Advanced Threat Detection** — threat scoring and some rate limiting features are unavailable.

Plan your password policy and rate limiting strategy accordingly, relying on complexity requirements and proxy-layer rate limiting instead.

**Upgrade process.** Download FusionAuth packages on a connected machine and transfer them to the air-gapped environment. Out-of-band SQL migration scripts are available for environments where the FusionAuth application cannot run database migrations directly. See the [upgrade guide](/docs/operate/deploy/upgrade) for details.

**License renewal.** FusionAuth licenses must be updated before they expire to avoid loss of paid features. In air-gapped environments, license activation is performed manually through the Reactor tab in the Admin UI. Plan for license renewal in advance — see [Licensing](/docs/get-started/core-concepts/licensing) for details on your plan's terms.

**Outbound hosts to block.** If you are enforcing air-gap at the network level rather than physically isolating the host, block the following FusionAuth-specific hostnames: `reactor.fusionauth.io`, `metrics.fusionauth.io`, `license.fusionauth.io`.

## Production Deployment Checklist

### All Deployments

- [ ] MFA enabled for all admin accounts
- [ ] Admin Users assigned least-privilege roles
- [ ] Password policy configured: minimum length increased, complexity enabled
- [ ] Breached password detection enabled (if licensed and network-connected)
- [ ] Access token lifetime set to 15 minutes or less
- [ ] Refresh token usage policy set to `OneTimeUse`
- [ ] SSO session timeout configured
- [ ] Locked account message normalized in Theme to match generic login failure
- [ ] Forgot password flow verified to return uniform responses
- [ ] PKCE set to `Required` (or `Not required when using client authentication`) on all Applications using the Authorization Code grant
- [ ] CORS configured with explicit origins (if using browser-based API calls)
- [ ] API Keys scoped to individual Tenants and restricted to required endpoints
- [ ] Cryptographic key rotation schedule established
- [ ] Webhooks created AND events enabled at the Tenant level
- [ ] `user.login.failed` and `user.login.suspicious` events forwarded to SIEM
- [ ] Audit log monitored for unexpected administrative actions
- [ ] Webhook transport secured with HTTPS and signing keys

### Self-Hosted Additional

- [ ] TLS 1.2+ enforced on all client-facing traffic
- [ ] FusionAuth behind a reverse proxy, not directly exposed to the internet
- [ ] Reverse proxy restricts access to `/admin/` by source IP
- [ ] Proxy trust policy set to `Only Configured` with explicit proxy IPs
- [ ] Per-IP rate limiting on `/oauth2/` endpoints at the proxy layer
- [ ] Database connection encrypted (TLS via JDBC URL parameters)
- [ ] Database credentials changed from defaults
- [ ] FusionAuth Search bound to localhost or private network only
- [ ] `Strict-Transport-Security` and other security headers set in the proxy
- [ ] No duplicate `X-Frame-Options` header (FusionAuth already sets this)
- [ ] Runtime mode set to `production`
- [ ] Configuration files readable only by the FusionAuth service user
- [ ] Kickstart file removed or access-restricted after provisioning
- [ ] Upgrade and patching schedule established
- [ ] Database backup procedure in place and tested

### Multi-Tenant Additional

- [ ] Tenant resolution handled server-side in your Application
- [ ] OAuth error template customized to remove `oauthJSONError` output
- [ ] Error page Theme applied to default Tenant
- [ ] Subdomain isolation or equivalent anti-enumeration pattern in place

### Kubernetes Additional

- [ ] Database credentials stored in Kubernetes Secrets (not Helm values)
- [ ] External secrets manager integrated (Vault, AWS Secrets Manager, or equivalent)
- [ ] Network policies restrict ingress and egress to FusionAuth pods
- [ ] Pod security context applied (drop capabilities, no privilege escalation)
- [ ] TLS terminated at Ingress with automated certificate management
- [ ] Database accessible only via private networking

### Air-Gapped Additional

- [ ] Password policy compensates for unavailable breached password detection
- [ ] Proxy-layer rate limiting compensates for unavailable ATD features
- [ ] License renewal procedure documented and scheduled
- [ ] Offline upgrade package distribution process established
