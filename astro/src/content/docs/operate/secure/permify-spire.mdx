## Overview

{/* this is a comment */}

[SPIFFE](https://spiffe.io/docs/latest/spiffe-about/overview) (Secure Production Identity Framework For Everyone) is a protocol that allows a microservice to authenticate itself to other microservices, without using a shared password or fixed network address.

[SPIRE](https://spiffe.io/docs/latest/spire-about/spire-concepts) is an open-source implementation (an actual application) of this protocol that provides the authentication certificates to the microservices.

Since SPIRE and SPIFFE are so closely related we'll use the terms interchangeably in this guide.

### How did auth work before SPIFFE?

Before the rise of cloud computing and Docker containers (**cloud native** systems), your software system might have been a database, some background jobs, and some web services running on a couple of corporate servers. How did a web service know if a background job had permissions to save information to the database? There were a few options:

- **Network rules**. The web service would allow any job that calls from a specific IP address on the network.
- **A password, API key, or SSH key**. The job would send a secret token with its request to the web service. (For SSH, the job would create a tunnel to the service, but the effect is the same.)
- **Mutual transport layer security (mTLS) X.509 certificates**. Similar to a password, except the job would first request a TLS certificate from a certification server using the password, then send certificate to the web service instead of the password.
- **Kerberos**. A complicated system, but basically the job would use its password to request a short-lived ticket from the Kerberos server, which the job would then pass to the web service, which would verify the ticket using the service's private password (which Kerberos already had).

### What are the problems with early auth?

You might have noticed the previous headings used the abbreviation **auth** and not **authentication** (proving identity) and **authorization** (proving permissions) separately. This hid the first problem: early systems often conflated authentication and authorization. Sending a password to a service provides authentication, but then the web service itself had to know what permissions that job had (authorization). It is possible for the web service to use the received username from the job to query an external authorization server, but many systems didn't reach this level of complexity.

Then the serious problems arise when you have a cloud system with hundreds of containers stopping and starting constantly with new IP addresses, across multiple networks. This is normal when using automated load balancing systems like Docker Swarm and Kubernetes. With such systems it is impossible to use network rules for authentication, since location is no longer static.

Container churn also causes speed problems. Recreating TLS certificates or requesting Kerberos authorization can be slow in older systems. Kerberos cryptography also relied on closely synchronized machine clocks. If clocks drifted, errors occurred. A new, robust, protocol was needed that can handle container boot authorization in milliseconds.

All the password-based authorization systems also have a security risk: storing the password on disk in a configuration file. If a hacker gains access to the password, she can use it to endlessly generate new security certificates or tickets. Similarly, certificates are generally long-lived. Companies maintained certificate revocation lists (CRLs), but if security compliance was poor, programmers would ignore the lists.

Trying to improve password security by instead using certificates is an example of the **secret zero** problem. No matter how many layers you use, you eventually need to hardcode a password somewhere that your service can access to obtain a certificate.

In summary, SPIFFE was designed to overcome the problems of:
- separating authorization and authentication,
- dynamic locations,
- storing secrets,
- long-lived authentication, and
- slow speeds.

### A brief SPIFFE example

Let's consider a brief SPIFFE example to learn how it solves the problems above.

Consider four components:
- The background job
- The web service the job wants to call
- A SPIRE server
- Two SPIRE agents (clients) that run on the same machines as the job and service.

Here's the process:
1. SPIRE agents start on all servers where services run. These agents are given the certificate of the SPIRE server and a way to call the server. This certificate is the root of all trust from now on.
1. An instance of the background job is started in a container by Docker Swarm to process a file.
1. The SPIRE agent running on the Docker host machine notices a new container has started, and checks the Docker service name, process ID, and user ID. The agent requests a **SVID** (SPIFFE Verifiable Identity Document), which is like a passport for the job, from the SPIRE server. The server returns an X.509 certificate (or JWT) extremely fast. The agent provides this certificate to the job via a local Unix socket. The SVID SPIFFE ID identifies the job with an identifier (name) like `spiffe://example.org/infrastructure/file-processor`. The agent also gives the service the SPIRE server certificate.
1. The job finishes executing and wants to send its result to the web service. The job calls the service, sending the service both the result data and the job's SVID (the actual certificate file read from disk).
1. When the service receives the request from the job, it can check the SVID validity internally using the SPIRE server certificate and any library that handles TLS cryptography (such libraries are available for any programming language). No call is needed to the SPIRE agent or server.
1. The service has now authenticated the job, but has not done authorization. SPIFFE doesn't handle authorization at all. So the service calls an authorization server (like Permify) to check if the job with ID `spiffe://example.org/infrastructure/file-processor` has permission to save its result.

### How does SPIFFE solve authentication problems?

You may have noticed some clear differences in this SPIFFE example compared to older authorization protocols:

- **No authorization**. SPIFFE handles only authentication. It completely ignores permissions a service might have. Permissions must be handled elsewhere. This mandates a clear separation of concerns.
- **Authentication by attestation**. Services don't have keep passwords anymore. Instead, a SPIFFE agent running on the same machine as the service (or on the Docker host) physically checks the process and user Ids of the service at the operating system level.
- **Speed**. After attestation, the agent will obtain a short-lived **SVID** (usually an X.509 certificate, but could be a JWT) for the service from the SPIFFE server very quickly. Refreshing an SVID is also fast. There is no need to contact the SPIFFE server to verify a SVID, so network use is reduced.
- **Location independence**. Since the SPIFFE agent runs on the same machine as the service, that machine can be anywhere on the network with any IP address. As long as the SPIFFE agent has been given the correct credentials for, and location of, the SPIFFE server, the agent can generate SVIDs for any services.
- **Short-lived SVIDs**. SVIDs are quick to generate, so there is no cost to refreshing them frequently. This eliminates the need for certificate revocation lists.

In summary, switching to SPIFFE frees your programmers from having to manage secrets, massively reduces the risk of a hacker gaining access to your network or (non-existent) service password, increases authentication speed and simplicity, and scales to any number of containers in any number of locations.

### SPIFFE terminology and some more detail

Now that you know the basics of SPIFFE, there are a few more details to know in order to understand the practical example that follows.

#### Workloads and services

With SPIFFE, services and jobs are called **workloads**. A workload might be a database, a web service, a bash script, or an entire external company's system wrapped in a single SVID. A workload is any instance of an app that does some work. (We need to say an app "instance" because a single app might be running dozens of times as different containers).

"Workload" is confusing at first because the word's standard meaning is a "set of tasks". So you can think of a workload as a "service" if you prefer. But technically, a workload is more varied than a service. A "service" implies other services can call it, which is not possible with a bash script, for example.

#### Humans aren't workloads (except when they are)

SPIRE attests the identity of a workload by verifying its details at the operating system level. This implies that any workload that needs a password to identify itself cannot be a workload. So for example, a human that logs in with a password cannot be given an SVID.

However, in addition to attestation, SPIFFE can be configured to accept and exchange tokens from other authentication providers. For example, a human could log in to any system that returns an OIDC certificate (like FusionAuth, Facebook, or KeyCloak), and then use that certificate to obtain an SVID from the SPIRE server. From then on, the human could pretend to be a workload, which might be useful for running debugging commands.

The possibilities here are complex: allowing multiple federated SPIRE servers, inter-company SPIRE trust, and chains of authentication between SPIRE and other identity providers. This article won't discuss these possibilities, but if your company needs flexible authentication, SPIFFE can probably handle it.

#### What can be a workload?

SPIRE can attest to the following types of workloads:

- Linux processes. Identified by user ID (UID), group ID (GID), or the file path of the binary (e.g., `/usr/bin/python3`).
- Docker containers: Identified by image ID, container labels, or environment variables.
- Kubernetes pods: Identified by namespace, service account, or pod labels.
- Systemd services: Any background service managed by systemd (like a database or web server), identified by their specific service unit name.
- Cloud instances: A whole VM in AWS, Azure, or GCP, usually identified by instance identity document (IID).
- AI agents (pending): Identified by specific metadata (capabilities and signed model hashes).

#### What external authentication providers work with SPIRE?

While SPIRE can't directly attest the workload types below, their authentication can be exchanged for SVIDs.

- OIDC — For example FusionAuth, and many other authentication gateways
- AWS Instance Identity Documents (IID)
- GCP Instance Identity Tokens (IIT)
- Azure Managed Service Identity (MSI)
- Kubernetes Service Accounts (PSAT)
- SSH keys
- Kerberos tickets

HashiCorp vault (a secret store) used to be just another workload accessible with SVIDs, but as of the end of 2025 it is now a SPIFFE server implementation itself (like SPIRE). Similar secrets services, OpenBao, Infiscal, and Conjur, can accept SVIDs but not generate them.

#### Control plane

Programmers comply with the separation of concerns principle by dividing code into layers for UI, logic, and data access. Similarly, system architecture can be divided into separate "planes". SPIRE works on the **control plane**, where it governs who can talk to whom. The data actually passed around by the workloads is part of the **data plane**.

## A working microservice example with SPIRE, Permify, Envoy, and Docker

To run the example system accompanying this guide, you need to have [Docker](https://docs.docker.com/engine/install). All components and services are run safely isolated inside Docker, and cannot access the files on the rest of your computer.

➡️ Use git to clone the example <a href={frontmatter.repository}>repository</a>, or download and unzip it.

➡️ Open a terminal in the repository directory and run the command below.

```sh
rm -rf ./spire/sockets/* && docker compose up
```

➡️ Once all containers are running, open a new terminal and run the commands below to test authentication and authorization from end to end.

```sh
curl http://localhost:7773/order  # allowed

curl http://localhost:7775/audit  # denied
```

The request to the orders service above triggers a call to the payments service. The payments service calls Permify with the SPIFFE ID of the orders service to see if orders is authorized to request a payment. Permify says orders has permission, so payments prints `allowed`.

The second request, to the audit service, also triggers a call to payments service. This time Permify does not find any permissions for audit to payments, and denies the request.

### Docker, Kubernetes, and cloud providers


Docker is the best way of showing a minimal example like this one — it's easy to run, and you can see exactly how each component is linked in the Compose file. However, most companies run SPIFFE in Kubernetes (through [Istio](https://istio.io)) or using the native identity services of the big three cloud providers (AWS, Azure, GCP). While Docker is simpler in general than the other options, it is actually more complicated to use for SPIFFE than Kubernetes:

- **Workload attestation**: Docker requires manual "join token" management, whereas Kubernetes uses automatic API-based attestation.
- **Request routing**: Services in Docker need manual path routing (Envoy cannot do this dynamically). Istio can handle transparent dynamic proxying from service to service.
- **Sockets**: Docker SPIRE agents need manual socket management for communication. Kubernetes doesn't need this.
- **Sidecars**: In Docker, each service needs an Envoy sidecar container with configuration file. Kubernetes doesn't need this.

Since you probably won't use Docker in production, this guide won't discuss configuring proxies in Envoy and Docker. You should rather spend time investigating Istio, or SPIFFE in your cloud provider, once you understand the fundamentals of SPIFFE presented here.

All containers are on the `spireNetwork` network so they can talk to each other.

The named volumes `serverBin` and `agentBin` expose the SPIRE binary files from the server and agent containers for other containers to use. Since the SPIRE containers do not have a shell (`sh` or `bash`), you need to interact with the containers (such as performing setup commands) from somewhere else. For example, the `spireServerInitialize` container uses the binary files from the server to register workloads. If the binary files were not exposed, you would have to install the SPIRE binaries on your host machine and run the setup commands from your host after starting Docker.

The `spireServer` container exposes two more volumes: 1) **a configuration file** and 2) **a socket**:

1. The `config/spireServer.conf` configuration file (shown below this bullet point) specifies a few basic settings:
  - `bind_address = "0.0.0.0"` ensures the server listens to requests from outside the container.
  - `socket_path` must match the socket path in the Compose file.
  - `DataStore` and `KeyManager` settings tell the server where to manage its data.
  - `NodeAttestor "join_token"` is important. This allows the SPIRE agent to register itself with the server, so it can begin authorizing workloads.

2. A socket is a shared file on disk processes can use to communicate. Unlike a TCP port, which any process can call, only a process with permission to a file can see it, making a socket more secure than a port. A process can ask also ask the operating system for details about the process on the other end of a socket, allowing SPIRE to perform attestation. The `registration.sock` socket is used by the `spireServerInitialize` container. This helper container's only job is to create registration entries in the server for the agent and workloads. The first two entries register the agent (with ID `spiffe://example.org/agent`) and create a "join token" — a secret password exposed in a shared volume that the SPIRE agent can give to the server to authenticate itself when starting. The next four registration entries for the services and Permify don't need a token, because the agent will authenticate them. Note that the SPIFFE ID domain name, like `example.org` matches the domain in server configuration file.

The SPIRE agent's configuration is simpler than the server, using only the command `command: ["run", "-config", "/etc/spire/agent.conf", "-joinTokenFile", "/opt/spire/sockets/agent_token"]`. This command configures the agent using `spireAgent.conf` (shown below) and authenticates the agent with the SPIRE server using the shared join token created earlier.

- `NodeAttestor "join_token"` tells the agent to use the join token to authenticate itself with the server.
- `insecure_bootstrap = true` tells the agent to trust the identity of the SPIRE server, which saves time in this example. You should **not** do this in production, as a malicious service could masquerade as the server. Rather pass the server's certificate into the agent at initialization manually.
