FusionAuth provides secure and functional authentication pages for your applications by default. But it takes more work to create registration and login pages that have excellent usability and delight new customers. This guide explains how to do that. You'll:
- Learn all the best practices for designing authentication pages.
- Learn what tools to use evaluate accessibility and standards compliance in your pages.
- Receive a summary checklist of the practices to use in designing or evaluating your own pages, including everything that's possible in the free version of FusionAuth.
- Be shown an example FusionAuth custom theme and walked through an evaluation of its merits and flaws.

## List Of Best Practices For Registration And Login

Let's start by reviewing everything good authentication pages should have. This generally comprises security, usability, accessibility, and attractiveness.

Registration and login pages should follow all the practices in the following subsections.

### ‚ÜîÔ∏è Separate

Both pages could be combined into a single page, after all they both contain just username and password fields. If the user is not yet registered when trying to sign in, why not automatically register them on the same page? The main reason is that separate pages avoid users creating duplicate accounts. People frequently forget which email addresses they used to register for a service (especially in recent years with an increased focus on privacy and email aliases), or mistype their address. Telling someone that their username does not exist is clearer than silently creating a new account. You also might want to include extra fields on a registration page, such as checkboxes for consent, which do not belong on a login page. Finally, if you want to protect users' privacy, having separate pages prevents attackers from entering email addresses into a page to see which users are registered for the service already.

### üßπ Minimal

Registration pages give the user work to do, which discourages them from using your service. Make that work as easy as possible by having pages with no clutter and as few fields to complete as possible. Autofocus the identifier textbox to save the user a click.

Make labels clickable and make the label focus its related textbox on click. This looks like `<label for="email">Email address</label><input id="email" >`.

Provide a toggleable eye icon so that users can view their password in plaintext, rather than having to retype the password in a confirmation textbox. Show the password requirements next to the password box so that the users don't have to invent a new password after the first one is rejected. Show validation for all fields on blur (when field loses focus) and not on form submission, for immediate feedback. If the page is refreshed (such as when form submission fails), the page should remember all entered values in every textbox, so the user doesn't have to retype anything.

Skip the login screen automatically and redirect the user to their account if they're already signed in.

Provide links between the registration and login pages so the user can quickly swap to the correct page if they started down the wrong path.

The reset password page should automatically fill the user's email address from the login page or browser cookie, to save the user from typing.

### ü¶Ø Accessible

This includes large readable fonts for those with poor eyesight, color-blindness, or dyslexia. Screen readers for the blind. Proper tab order and keyboard support for those who can't use a mouse. Ensure your site works correctly on desktop and mobile. Accessibility also means that users can actually load the page, so be careful with images that are too large, don't exclude those using a VPN, and have a backup host if your Cloudflare or AWS web host crashes. Show translated pages in the language of your users if appropriate.

Give clear and complete instructions for inexperienced users: "I don't know what the password is! Oh, I'm supposed to *invent* my own password? Why didn't you say so".

Show an indicator if the capslock key is on. Don't disable the paste ability on fields. Some users physically need it, and it's useful for those keep their passwords in a separate database.

Show a spinner for feedback for any requests to the server, such as password validation and form submission. Users can quickly lose interest and assume a page is broken if there's no visual feedback.

Make timeouts long. Even a short registration page could take a user a while to complete, especially if life interrupts them. It's annoying if when they finally click submit, the page rejects them due to an anti-forgery cookie timeout and they need to resubmit.

Label your email textbox with `type=email` in HTML to get the appropriate mobile keyboard layout for fast entry. It's faster to enter an email address if you don't have to go hunt for the `@` sign on your keyboard. Do the same for telephone numbers (`type=tel`), URLs (`type=url`), and numbers (`type=number`).

Use the `autocomplete="username"` and `autocomplete="current-password"` attributes on fields so the browser can save the user from typing.

If your pages uses icons (like a little envelope picture in the email field), ensure they have `aria-hidden="true"` if they are purely decorative. If instead the icons are meaningful, provide an `alt` text description for screen readers.

Use an ARIA live region, like `<div role="alert">Incorrect password</div>`, so that screen readers will be aware when the div content changes dynamically through JavaScript (as opposed to reading the page only the first time on load).

### üåê Standards Compliant

This includes HTML5, OAuth 2, email, and cross-browser compatibility with functions like autocomplete and password managers.

Allow all legal email addresses. For example, `+` and `/` are legal characters in addresses, as are two-letter domain names, but are frequently banned by persnickety registration pages.

### üîê Secure

When using FusionAuth, this is the easiest part. The hosted login pages are the result of a company dedicated to authentication and are secure by default. The tricky part is deciding whether to support login with email addresses, usernames, or mobile phones. And whether to use passkeys, single sign-on (SSO) or social login, and two-factor authentication. Each of these choices has security and ease-of-use implications. You need to decide what is best for the majority of your users. Passwords should follow modern best practices, like [zxcvbn](https://github.com/zxcvbn-ts/zxcvbn), and not superstition about uppercase and special characters.

Use CAPTCHA to ensure only humans, and not spam bots, attempt to register and log in. Prevent brute-force password attacks by rate limiting login attempts. After too many incorrect passwords lock a user's account for a minute.

Upgrade your FusionAuth instance frequently to get all security patches. Use HTTPS and not HTTP for connections.

Provide a "Stay logged-in on this device" checkbox to allow the user to explicitly say whether they want the convenience of avoiding future logins on their home computer, or the safety of staying logged out on a shared public computer. Provide a clear log out button once logged in.

Ensure that your client-side and server-side validation are identical. If client is stricter, an attacker could bypass the HTML page to send a form directly to the server, where the server would accept unsafe fields. If the server is stricter, a user could enter a valid password on the registration page, only for it to be rejected on the server at submission.

### ‚öñÔ∏è Legal And Ethical

You need to be compliant with GDPR and the Cookie law if your service is available to any European citizen. Put links to privacy and data use policies on your registration page. Include opt-in checkboxes for any personal data collection. Obtain consent for any non-essential cookies. For more information on this complex topic, see the blog articles on [EU laws](todo get link when FusionAuth publishes this) and [configuring FusionAuth to comply with GDPR](todo get link when FusionAuth publishes this).

### üëÅÔ∏è Able To Collect Metrics

Your registration page should track, in a way that doesn't collect personal data, how many people attempt, complete, and abandon registration. If possible, track where users abandon registration to learn what might be discouraging them and what you can improve.

You also might want to use A/B testing with multiple versions of a registration page to see which is the most convenient.

### üåº Good Looking

A page that is visually well-designed persuades users that your business is successful, professional, and trustworthy. Elegant animations often delight users. But this isn't about only layout and styling. Typography matters, as does your text itself. The language on the page must be clear, grammatically correct, and the tone should match your business. Ensure translations to foreign languages are correct too.

Follow the user's choice for dark mode or light mode in their browser or operating system settings. Showing a dazzling white registration page to users who prefer dark mode will make them immediately dislike your app.

## FusionAuth Theming Options

If you read the FusionAuth [theming overview](/docs/customize/look-and-feel), you'll know there are three ways to customize your authentication pages:
- use the [Simple Theme Editor](/docs/customize/look-and-feel/simple-theme-editor) to make small changes to one of the default FusionAuth themes.
- use the [Advanced Theme Editor](/docs/customize/look-and-feel/advanced-theme-editor) to overhaul everything with custom CSS, JS, and the [Java FreeMarker template language](https://freemarker.apache.org/).
- write custom pages in your own app that call the [FusionAuth API for authentication](/docs/apis/authentication), and don't use the hosted login pages. (This is dangerous as it's easy to make security mistakes.)

The FusionAuth default theme is minimal and secure, but fails to follow many of the best practices in the previous table. Besides specific flaws like not having a viewable password, not validating on blur, and not providing clear instructions and password requirements, the theme is also visually bland and unappealing. It's designed purely as a placeholder that you should replace with your own branding. You can see how the theme looks in the screenshots below.

![Default registration](/img/docs/customize/look-and-feel/best-practices/defaultRegistration.webp)

![Default login](/img/docs/customize/look-and-feel/best-practices/defaultLogin.webp)

![Default forgot password](/img/docs/customize/look-and-feel/best-practices/defaultForgot.webp)

If you think the default theme has adequate usability for your app, and want to make only a few visual adjustments, you can use the simple theme editor. But to follow as many best practices as possible, you'll need the advanced theme editor, which allows for custom JavaScript.

<Aside type="caution">
If you start creating your own custom simple theme and later decide to switch to a custom advanced theme, there is no simple path to switch. If you try to download a simple theme for customization instead of an advanced theme, the files will be empty. You have to duplicate an advanced theme and manually customize it to match the changes you made to the simple theme.
</Aside>

## Evaluating A Custom Advanced Theme

While the default FusionAuth theme is simplistic, there are four impressive themes available at [FusionAuth Themes](https://www.fusionauththemes.com) for you to use as a template when creating your own. In this section, the best practice checklist is used to evaluate the first theme, for Iron Pixel, to show you exactly what you should be aware of when designing your page.

You can view the theme with registration and login pages at the mock [Iron Pixel website](https://iron-pixel.fusionauththemes.com). Browse around the site to see if you can spot any flaws yourself before continuing.

![Iron Pixel theme](/img/docs/customize/look-and-feel/best-practices/ironpixelTheme.webp)

The addendum at the bottom of this article provides step by step instructions to install the Iron Pixel theme on your own instance of FusionAuth. This is a convenient base to start building your own theme from, but isn't necessary to do to understand the evaluation below.

The list below holds every best practice from the checklist, and evaluates the authentication pages of the sample site against them. Practices followed are marked with ‚úÖ and places the site need improvement are marked with ‚úñÔ∏è.

- ‚úÖ Separate pages. Links here look like tabs. Clear and fast.
- ‚úñÔ∏è Hide registered accounts. For privacy, an attacker should never be told if an email for a service is in use. This is not possible in FusionAuth and has been a [complaint since 2023](/community/forum/topic/2484/registering-with-an-existing-email).
  ![Email exists](/img/docs/customize/look-and-feel/best-practices/emailExists.webp)
- ‚úñÔ∏è Minimal. At first, the page might look minimal, but interactions show it lacks attention to detail and hasn't been tested. For example, if your password at login is incorrect, the error is shown in two places and the toast moves the entire page, instead of floating above it. This clutter should be removed.
  ![Invalid password clutter](/img/docs/customize/look-and-feel/best-practices/invalidClutter.webp)
  Additionally, on the forgot password page, there is an unnecessary person logo and unstyled spacing. These small flaws say a lot about the lack of the attention to detail of a business.
  ![Unstyled forgot password page](/img/docs/customize/look-and-feel/best-practices/forgotPasswordUgly.webp)
  Finally, you can see that "Sign up" is shown twice, once on the tab at the top and once at the bottom of the page.
- ‚úÖ Autofocus first textbox
- ‚úñÔ∏è Clickable labels. The username and password fields in this theme don't have labels. Instead, they use the hint text when the fields are blank. This is good for minimalism, but not good for clarity, because the hint disappears when the user starts typing. If your users are technically experienced, this is fine, but if you're aiming for maximum clarity, you need to add field labels.
- ‚úñÔ∏è Viewable password. In this theme there is neither a password confirmation box nor a way to view the password you entered. You need at least one, especially when on mobile where it's easy to mistype.
- ‚úñÔ∏è Show password requirements. This instruction has to be added manually to a registration page. Be careful doing this, because an administrator might change the requirements in the FusionAuth backend, but forget to update the text in the theme frontend.
- ‚úñÔ∏è Validate on blur. Currently, validation happens only at form submission.
- ‚úÖ Remember entries. The only field here is for an email address, and that is autofilled if form submission fails and the page refreshes. The password field should not autofill on failure, and does not in this theme.
- ‚úÖ Skip login. This functionality is easy to implement with FusionAuth and is handled by your app, not FusionAuth directly. For more details, see one of the [Quickstarts](/docs/quickstarts/).
- ‚úÖ Links between pages. It appears as if the page uses tabs, but these are actually links that reload a similar looking page.
- ‚úñÔ∏è Autofilled reset password username
- ‚úÖ Large fonts, clear contrast. This theme has both, and responds correctly to page zoom.
- ‚úÖ Dyslexic font. The theme uses standard and readable fonts, which is appropriate for most web pages. If you were creating a service for those with special reading needs, you could consider switching to a typeface like [OpenDyslexic](https://opendyslexic.org).
- ‚úñÔ∏è Colorblind safe. You can use a website like [Pilestone](https://pilestone.com/pages/color-blindness-simulator) to see how an image appears with various forms of colorblindess.
  ![Colorblind](/img/docs/customize/look-and-feel/best-practices/colorblind.webp)
  However, the simplest test is to remove color all together and preview your site in black and white. If any information is conveyed exclusively by color, it should be improved. For example, on the login page the invalid password message is red but with no exclamation symbol, and so wouldn't appear as a problem to a colorblind user. To preview an entire site in black and white, use your browser developer tools. In Chromium-based browsers, you can find the "rendering" setting from the "more tools" option.
  ![Rendering tools](/img/docs/customize/look-and-feel/best-practices/renderingTools.webp)
  As well as previewing color problems, you can also preview your site blurred or at reduced contrast. In the image below you can see a user with contrast issues would struggle to read the field hint texts and they should be improved ‚Äî something already noted earlier in the clickable labels practice.
  ![Reduced contrast](/img/docs/customize/look-and-feel/best-practices/reducedContrast.webp)
  The Firefox accessibility tab is faster to access than Chromium's, and also shows you tab order and missing field descriptions.
  ![Firefox preview](/img/docs/customize/look-and-feel/best-practices/firefoxCheck.webp)
- ‚úñÔ∏è Blind safe. Testing full blindness and using a screen reader is more complicated than testing color blindness. Screen readers are configured at the operating system level and differ between Linux, MacOS, and Windows. For a partial test, you can use the [Silktide Accessibility Checker](https://chromewebstore.google.com/detail/silktide-accessibility-ch/mpobacholfblmnpnfbiomjkecoojakah). The free Silktide plugin offers accessibility checking even more powerful than Firefox, allowing to find a variety of accessibility problems.
  ![Silktide](/img/docs/customize/look-and-feel/best-practices/silktide.webp)
  Stepping through the login page using the virtual screen reader immediately highlights problems. The email textbox not only has no label, but also no description at all for blind users.
  ![Screen reader](/img/docs/customize/look-and-feel/best-practices/screenReader.webp)
- ‚úÖ Tab order.
- ‚úÖ Keyboard only. Nothing on this page needs the mouse. However, you need to do additional checks for keyboard use if you use 3rd party plugins, like CAPTCHA.
- ‚úñÔ∏è Mobile & desktop. You can preview a mobile layout of your page using most browsers, and extensions like Silktide. You can see in the screenshots below that the theme developers sacrificed the screenshots and videos of their game instead of moving them below the form or adding them as a third tab. No one can say whether this design choice is good or bad without user testing. However, there is an indisputable design error on mobile ‚Äî the margins of the page are different for each tab. This is caused by the top and bottom div margin automatically adjusting to tab content. While vertical space wasn't a problem on desktop, on mobile the layout bug is quickly exposed.
  ![Mobile login](/img/docs/customize/look-and-feel/best-practices/loginMobile.webp)
  ![Mobile register](/img/docs/customize/look-and-feel/best-practices/registerMobile.webp)
- ‚úÖ Few KB. To test page load speed, you can choose data throttling (shown at the top of the screenshot above). This theme loads quickly, and is a point in favor of removing the large game screenshots shown in the desktop version of the site. The images on the page use `loading="lazy"`, which tells the browser not to load the image from the server until the user can see it. This saves bandwidth on the mobile version as the image will never be visible.
- ‚úÖ VPN safe ¬∑ Country restrictions. To set allowed IP ranges in FusionAuth, browse to <Breadcrumb>Settings -> IP Access Control</Breadcrumb>. The default is no restrictions, which is fine.
- ‚úÖ Backup host. FusionAuth is stateless and you can host multiple instances pointing to the same database without a problem.
- ‚úÖ Language translations. This theme has a language selector at the bottom of the page but is currently available only in English. FusionAuth themes support multiple languages. You should show the language selector at the top of the page though so users don't miss it, and consider adding a flag icon for those who don't understand the word "English".
- ‚úñÔ∏è Clear instructions. The purpose of the page is clear, but some controls are not labelled, especially the most important: email and password. Add instruction labels "Enter your email address" and "Create a password".
- ‚úñÔ∏è Capslock indicator. The page doesn't have one, which is even more dangerous since there is no way to view the password nor a textbox to confirm it.
- ‚úÖ Paste allowed.
- ‚úñÔ∏è Feedback spinner. No feedback is given when the <InlineUIElement>Sign Up</InlineUIElement> button is clicked. If you enable bandwidth throttling in browser tools, you can see how confusing this is. After attempting to sign up, the user can see no evidence that the button was clicked or that the service is processing anything. The page looks dead, and remains so for about five seconds ‚Äî easily enough time for a user to abandon registration and close the page.
- ‚úÖ Long timeouts. By default, FusionAuth gives you 30 minutes to log in before the page times out. To adjust this, browse in the FusionAuth admin UI to <Breadcrumb>Tenants -> Edit your tenant -> Advanced tab</Breadcrumb> and change the value of <InlineField>Login timeout</InlineField>, then save the tenant.
- ‚úñÔ∏è `type='email'`. While the password textbox correctly have the type `password`, the email textbox incorrectly uses the `text` type. Other attributes, like `autofocus` and `autocomplete` are correctly set: `<input id="loginId" type="text" autocapitalize="none" autocomplete="on" autocorrect="off" spellcheck="false" autofocus="autofocus" placeholder="Email">`
- ‚úÖ `autocomplete`
- ‚úñÔ∏è `aria-hidden` ¬∑ `alt`. No images on the page are purely decorative so `aria-hidden` is not needed. But the screenshots do not have descriptions in `alt` attributes for screen readers (and machine understanding for SEO and LLM understanding of your site).
- ‚úÖ `aria-live`. Since password validation is done only on form submission and not dynamically through JavaScript, not dynamic screen reader update is needed here.
- ‚úñÔ∏è Standards compliance. To continuously test your page for HTML and CSS compliance you can use a tool like the [Nu HTML Checker](https://github.com/validator/validator) locally. For a single test, you can use the [online version](https://validator.nu). As you can see below, the Iron Pixel theme contains numerous instances of invalid HTML and CSS that should be fixed.
  ![Nu HTML Checker](/img/docs/customize/look-and-feel/best-practices/nu.webp)
  FusionAuth handles unusual email addresses without a problem, including names with quotes, spaces, special characters, and Unicode text.
- ‚úÖ Cross-browser compliance. The site appears correct in Firefox and Chromium-based browsers, but thorough testing is very difficult. Safari runs only on expensive Apple hardware and your local computer cannot install multiple versions of the same browser. If supporting as many browsers as possible is vital, you need to use a paid multi-browser testing service.
- ‚úÖ Multiple authentication: username, email, phone, social media, passkeys. FusionAuth supports logging in with one of: username, phone, or email (which includes email supplied by external authentication providers like Google). In this example, email was chosen and social login through game services was also provided. Users appreciate this flexibility.
- ‚úÖ Two-factor authentication is not used on this site. This is neither good nor bad. Presumably, a gaming site does not hold sensitive data and so chose ease of use over extra security.
- ‚úñÔ∏è Custom password rules. While not being overly strict with password makes life simpler for users, this site currently allows a password of `password`, which is not secure enough. If you want to avoid forcing the user to use special characters, consider increasing the password length requirement to 24 characters to encourage the user to enter a phrase.
  ![Password rules](/img/docs/customize/look-and-feel/best-practices/passwordRules.webp)
- ‚úñÔ∏è CAPTCHA. Since this site is just an example, it does not use CAPTCHA, which needs a paid provider and a paid FusionAuth license. However, in reality using CAPTCHA is unavoidable. If you don't have it, your site will quickly be overwhelmed with spam accounts or denial of service attacks from competitors. Be careful which provider you choose. Users frequently express frustration with Google's "Click all the bicycles" test when they fail for no apparent reason. Some users prefer hCAPTCHA's puzzles as straightforward, but some users find them even more onerous than Google. While invisible CAPTCHA (which invisibly observes users' behavior on the page instead of showing a puzzle) seems like ideal usability, if it rejects a human user incorrectly there is no remedy.
  ![CAPTCHA](/img/docs/customize/look-and-feel/best-practices/captcha.webp)
- ‚úñÔ∏è Rate-limit incorrect passwords. By default, FusionAuth doesn't rate limit anything. This can be configured easily in the <Breadcrumb>Security</Breadcrumb> tab of the tenant.
  ![Rate limit](/img/docs/customize/look-and-feel/best-practices/rateLimit.webp)
- ‚úÖ HTTPS
- ‚úÖ Stay logged-in
- ‚úÖ Log out.
- ‚úñÔ∏è Matching server/client validation. As mentioned earlier, validation is not performed client-side at all in this design.
- ‚úñÔ∏è Consent checkboxes ¬∑ Cookie consent banner ¬∑ Disable cookies if no consent. No consent is requested anywhere on sign up page. There is also no link to any privacy policy or data use policy.
- ‚úñÔ∏è Registration metrics. No client-side monitoring is done on this page. It can be added with JavaScript.
- ‚úñÔ∏è A/B testing. This is not possible in FusionAuth. Only one version of a theme and each type of authentication page is supported per application and per tenant. ## Addendum: How To Create A Custom Advanced Theme From The FusionAuth Themes Templates

To create a custom theme (simple or advanced), first use the FusionAuth admin UI to duplicate the FusionAuth theme template in <Breadcrumb>Customizations -> Themes</Breadcrumb> and give it a UUID, for example `dc243231-0808-497a-9938-2d2a35860e01`.

Choose that theme in the dropdown of the <InlineUIElement>General</InlineUIElement> tab when editing the <Breadcrumb>Tenant</Breadcrumb> to apply it to all your applications.

Now you have your own theme to work with. In order to customize it, you need to download all the files to a local directory, edit them in a text editor, and upload them. This process is simplified by the FusionAuth Theme Helper tool. Run the commands below in a terminal to use it (you need Git and Node).

```sh
git clone https://github.com/FusionAuth/fusionauth-theme-helper
cd fusionauth-theme-helper
cp .env.sample .env
vi .env # set your FusionAuth key from Settings -> API Keys

# Example:
#API_KEY=33052c8a-c283-4e96-9d2a-eb1215c69f8f-not-for-prod
#THEME_ID=dc243231-0808-497a-9938-2d2a35860e01
#FUSIONAUTH_URL=http://localhost:9011
#TMP_DIR=tmp

# save and exit vi

npm install
./download.sh
```

You should now have a copy of the custom theme files in the `fusionauth-theme-helper/tmp` directory.

Once you've made your changes, run `./upload.sh` to save your changes back to FusionAuth so you can see what they look like.

If you want to use Iron Pixel as a template, download the files from https://github.com/FusionAuth/fusionauth-themes/tree/main/packages/fusionauth/iron-pixel. You should have a directory containing `input.css` and a `tpl` directory ("TemPLate"). Copy the contents of `tpl` up to the same level as `input.css`. Then copy all the files into the `tmp` folder that the FusionAuth Theme Helper download script created. You now have a copy of Iron Pixel, ready for editing and uploading.
