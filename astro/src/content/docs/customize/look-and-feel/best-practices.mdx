FusionAuth provides secure and functional authentication pages for your applications by default. But it takes more work to create registration and login pages that have excellent usability and delight new customers. This guide explains how to do that, covering:
- The best practices for designing authentication pages
- Tools for evaluating accessibility and standards compliance in your pages
- A summary checklist for designing or evaluating your own pages, including everything that's possible in the free version of FusionAuth
- A demo evaluation of an example FusionAuth custom theme

## List Of Best Practices For Registration And Login Pages

Let's start by reviewing everything good authentication pages should have. This generally comprises security, usability, accessibility, and attractiveness.

Registration and login pages should follow all the practices in the following subsections.

### ‚ÜîÔ∏è Separate

Both pages could be combined into a single page. After all, they both contain just username and password fields. If the user has not yet registered when trying to sign in, why not automatically register them on the same page? The main reason is that having separate pages prevents users from creating duplicate accounts. People frequently forget which email addresses they used to register for a service (especially with the increased focus on privacy and email aliases in recent years) or mistype their address. Telling someone that their username does not exist is clearer than silently creating a new account. You may also want to include extra fields on a registration page, such as checkboxes for consent, which do not belong on a login page. Finally, if you want to protect users' privacy, having separate pages prevents attackers from entering email addresses in the login field to check which users are registered for the service.

### üßπ Minimal

Registration pages give the user work to do, which discourages them from using your service. Make that work as easy as possible by designing pages with no clutter and as few fields to complete as possible:

- Autofocus the identifier textbox to save the user a click.
- Make labels clickable, and on a click, make the label focus its related textbox. This looks like `<label for="email">Email address</label><input id="email" >`.
- Provide a toggleable eye icon so that users can view their password in plaintext, rather than having to retype the password in a confirmation textbox. 
- Show the password requirements next to the password box, so they don't have to invent a new password after the first one is rejected.
- Show validation for all fields on blur (when a field loses focus) and not on form submission, for immediate feedback. 
- If the page is refreshed (such as when form submission fails), the page should remember the values entered in every textbox, so the user doesn't have to retype anything.
- Skip the login screen automatically and redirect the user to their account if they're already signed in.
- Provide links between the registration and login pages so the user can quickly swap to the correct page if they start down the wrong path.
- The reset password page should automatically enter the user's email address from the login page or browser cookie to save the user from typing.

### ü¶Ø Accessible

For accessibility, use large, readable fonts for those with poor eyesight, color-blindness, or dyslexia. Add screen readers for the blind, and proper tab order and keyboard support for those who can't use a mouse. Ensure your site works correctly on desktop and mobile.

Accessibility also means that users can actually load the page, so be careful with images that are too large, don't exclude anyone using a VPN, and have a backup host if your Cloudflare or AWS web host crashes. Show translated pages in the language of your users if appropriate.

Give clear and complete instructions for inexperienced users: "I don't know what the password is! Oh, I'm supposed to *invent* my own password? Why didn't you say so?"

Show an indicator if the Caps Lock key is on. Don't disable the paste ability on fields. Some users physically need it, and it's useful for those who keep their passwords in a separate database.

Show a spinner to provide feedback for any requests to the server, such as password validation and form submission. Users quickly lose interest and assume a page is broken if there's no visual feedback.

Make timeouts long. Even a short registration page could take a user a while to complete, especially if life interrupts them. It's annoying if, when a user finally clicks submit, the page rejects them due to an anti-forgery cookie timeout, and they need to resubmit.

Label your email textbox with `type=email` in HTML to get the appropriate mobile keyboard layout for fast entry. It's faster to enter an email address if you don't have to hunt for the `@` sign on your keyboard. Do the same for telephone numbers (`type=tel`), URLs (`type=url`), and numbers (`type=number`).

Use the `autocomplete="username"` and `autocomplete="current-password"` attributes on fields so the browser can save the user from typing.

If your pages use icons (like a little envelope picture in the email field), ensure the icons have `aria-hidden="true"` if they are purely decorative. If instead the icons are meaningful, provide an **alt text** description for screen readers.

Use an ARIA live region, like `<div role="alert">Incorrect password</div>`, so that screen readers are aware of when the division content changes dynamically through JavaScript (as opposed to reading the page only the first time on load).

### üåê Standards Compliant

Ensuring compliance with standards includes checking HTML5, OAuth 2, email, and cross-browser compatibility with functions like autocomplete and password managers.

Allow all legal email addresses. For example, `+` and `/` are legal characters in addresses, as are two-letter domain names, but they are frequently banned by persnickety registration pages.

### üîê Secure

When using FusionAuth, this is the easiest part. The hosted login pages are the result of a company dedicated to authentication and are secure by default.

The tricky part is deciding whether to support login with email addresses, usernames, or mobile phones, and whether to use passkeys, single sign-on (SSO), or social login, and two-factor authentication. Each of these choices has security and ease-of-use implications. You need to decide what is best for the majority of your users. Passwords should follow modern best practices, like [zxcvbn](https://github.com/zxcvbn-ts/zxcvbn), not superstitions about uppercase and special characters.

Use CAPTCHA to ensure that only humans, and not spam bots, attempt to register and log in. Prevent brute-force password attacks by rate-limiting login attempts. After the user enters too many incorrect passwords, lock their account for a minute.

Upgrade your FusionAuth instance frequently to get all security patches. Use HTTPS, not HTTP, for connections.

Provide a **Stay logged in on this device** checkbox to allow the user to explicitly say whether they want the convenience of avoiding future logins on their home computer, or the safety of staying logged out on a shared public computer. Provide a clear **Log out** button once they have logged in.

Ensure that your client-side validation and server-side validation are identical. If the client is stricter, an attacker could bypass the HTML page to send a form directly to the server, where the server would accept unsafe fields. If the server is stricter, a user could enter a valid password on the registration page, only for it to be rejected on the server at submission.

### ‚öñÔ∏è Legal And Ethical

You need to be compliant with GDPR and the cookie law if your service is available to any European citizen. Put links to privacy and data-use policies on your registration page. Include opt-in checkboxes for any personal data collection. Obtain consent for any non-essential cookies. For more information on this complex topic, see the blog articles on [EU laws](todo get link when FusionAuth publishes this) and [configuring FusionAuth to comply with GDPR](todo get link when FusionAuth publishes this).

### üëÅÔ∏è Able To Collect Metrics

Your registration page should track, in a way that doesn't collect personal data, how many people attempt, complete, and abandon registration. If possible, track where users abandon registration to learn what might be discouraging them and what you can improve.

You may also want to use A/B testing with multiple versions of a registration page to see which is the most convenient.

### üåº Good Looking

A page that is visually well-designed persuades users that your business is successful, professional, and trustworthy. Elegant animations often delight users. But this isn't about only layout and styling. Typography matters, as does your text itself. The language on the page must be clear and grammatically correct, and the tone should match your business. Ensure translations to foreign languages are correct too.

Follow the user's choice for dark mode or light mode in their browser or operating system settings. Showing a dazzling white registration page to users who prefer dark mode will make them immediately dislike your app.

## FusionAuth Theming Options

The FusionAuth [theming overview](/docs/customize/look-and-feel) covers three ways you can customize your authentication pages:

- Use the [Simple Theme Editor](/docs/customize/look-and-feel/simple-theme-editor) to make small changes to one of the default FusionAuth themes.
- Use the [Advanced Theme Editor](/docs/customize/look-and-feel/advanced-theme-editor) to overhaul everything with custom CSS, JavaScript, and the [Java FreeMarker template language](https://freemarker.apache.org/).
- Write custom pages in your own app that call the [FusionAuth API for authentication](/docs/apis/authentication), instead of using the hosted login pages. (This is dangerous, as it's easy to make security mistakes.)

The FusionAuth default theme is minimal and secure, but doesn't follow all the best practices in the previous table. As the following screenshots show, it's designed as a placeholder that you can replace with your own branding.

![Default registration](/img/docs/customize/look-and-feel/best-practices/defaultRegistration.webp)

![Default login](/img/docs/customize/look-and-feel/best-practices/defaultLogin.webp)

![Default forgot password](/img/docs/customize/look-and-feel/best-practices/defaultForgot.webp)

If the default theme has adequate usability for your app, and you want to make only a few visual adjustments, use the Simple Theme Editor. 

If you want to follow as many best practices as possible, use the Advanced Theme Editor. The advanced editor allows for custom JavaScript, so you can add a viewable password, validate fields on blur, and provide clear instructions and password requirements.

<Aside type="caution">
If you start creating your own custom simple theme and later decide to switch to a custom advanced theme, there is no simple path for switching. If you try to download a simple theme for customization instead of an advanced theme, the files will be empty. You have to duplicate an advanced theme and manually customize it to match the changes you made to the simple theme.
</Aside>

## Evaluating A Custom Advanced Theme

While the default FusionAuth theme is simplistic, there are four impressive themes available at [FusionAuth Themes](https://www.fusionauththemes.com) for you to use as a template when creating your own. This section uses the best practices checklist evaluate the first theme, Iron Pixel, showing you exactly what to be aware of when designing your page.

You can view the theme with registration and login pages at the mock [Iron Pixel website](https://iron-pixel.fusionauththemes.com). Browse around the site to see if you can spot any flaws yourself before continuing.

- ‚úÖ **Separate Pages:** Links here look like tabs. Clear and fast.
- ‚úñÔ∏è **Hide Registered Accounts:** For privacy, an attacker should never be told if an email for a service is in use. This is not possible in FusionAuth and has been a [complaint since 2023](/community/forum/topic/2484/registering-with-an-existing-email).
  ![Email exists](/img/docs/customize/look-and-feel/best-practices/emailExists.webp)
- ‚úñÔ∏è **Minimal:** At first, the page looks minimal, but interactions show it lacks attention to detail and hasn't been tested. For example, if your password at login is incorrect, the error is shown in two places, and the toast moves the entire page instead of floating above it. This clutter should be removed.
  ![Invalid password clutter](/img/docs/customize/look-and-feel/best-practices/invalidClutter.webp)
  Additionally, on the Forgot Password page, there is an unnecessary person logo and unstyled spacing. Such small flaws reveal a business's lack of attention to detail.
  ![Unstyled forgot password page](/img/docs/customize/look-and-feel/best-practices/forgotPasswordUgly.webp)
  Finally, the page displays **Sign up** twice, once on the tab at the top and once at the bottom.
- ‚úÖ **Autofocus First Textbox**
- ‚úñÔ∏è **Clickable Labels:** The username and password fields in this theme are not labelled. Instead, they use the hint text when the fields are blank. This is good for minimalism, but not good for clarity, because the hint disappears when the user starts typing. If your users are technically experienced, this is fine, but if you're aiming for maximum clarity, you need to add field labels.
- ‚úñÔ∏è **Viewable Password:** This theme has neither a password confirmation box nor a way to view the password you entered. You need at least one, especially on mobile, where it's easy to mistype.
- ‚úñÔ∏è **Show Password Requirements:** You need to add this instruction manually to a registration page. Be careful doing this, because an administrator might change the requirements in the FusionAuth backend but forget to update the text in the theme frontend.
- ‚úñÔ∏è **Validate On Blur:** Currently, validation happens only at form submission.
- ‚úÖ **Remember Entries:** The only field here is for an email address, which is autofilled if form submission fails and the page refreshes. The password field should not autofill on failure, and does not in this theme.
- ‚úÖ **Skip Login:** This functionality is easy to implement with FusionAuth and is handled by your app, not FusionAuth directly. For more details, see one of the [quickstarts](/docs/quickstarts/).
- ‚úÖ **Links Between Pages:** The page appears to use tabs, but these are actually links that reload a similar-looking page.
- ‚úñÔ∏è **Autofilled Reset Password Username**
- ‚úÖ **Large Fonts, Clear Contrast:** This theme has both, and responds correctly to page zoom.
- ‚úÖ **Dyslexic Font:** The theme uses standard and readable fonts, which are appropriate for most web pages. If you were creating a service for those with special reading needs, you could consider switching to a typeface like [OpenDyslexic](https://opendyslexic.org).
- ‚úñÔ∏è **Colorblind Safe:** You can use a website like [Pilestone](https://pilestone.com/pages/color-blindness-simulator) to see how an image appears with various forms of colorblindness.
  ![Colorblind](/img/docs/customize/look-and-feel/best-practices/colorblind.webp)
  However, the simplest test is to remove color altogether and preview your site in black and white. If any information is conveyed exclusively by color, it should be improved. For example, on the login page, the invalid password message is red, but with no exclamation symbol, and so it doesn't appear as a problem to a colorblind user. To preview an entire site in black and white, use your browser developer tools. In Chromium-based browsers, you can find the **Rendering** setting from the **More tools** option.
  ![Rendering tools](/img/docs/customize/look-and-feel/best-practices/renderingTools.webp)
  As well as previewing color problems, you can also preview your site blurred or at reduced contrast. The following image shows that a user with contrast issues would struggle to read the field hint texts, which should be improved (as noted in the Clickable Labels practice).
  ![Reduced contrast](/img/docs/customize/look-and-feel/best-practices/reducedContrast.webp)
  The Firefox accessibility tab is faster to access than Chromium's, and also shows you tab order and missing field descriptions.
  ![Firefox preview](/img/docs/customize/look-and-feel/best-practices/firefoxCheck.webp)

The free Silktide plugin offers accessibility checking even more powerful than Firefox, allowing you to find a variety of accessibility problems.
  ![Silktide](/img/docs/customize/look-and-feel/best-practices/silktide.webp)

- ‚úÖ **Tab Order**
- ‚úÖ **Keyboard Only:** Nothing on this page needs the mouse. However, you need to do additional checks for keyboard use if you use third-party plugins, like CAPTCHA.
- ‚úñÔ∏è **Mobile & Desktop:** You can preview a mobile layout of your page using most browsers and extensions like Silktide. This saves bandwidth on the mobile version because the image will never be visible.

- ‚úÖ **Backup Host:** FusionAuth is stateless. You can host multiple instances that point to the same database without a problem.
- ‚úÖ **Language Translations:** This theme has a language selector at the bottom of the page but is currently available only in English. FusionAuth themes support multiple languages. Move the language selector to the top of the page so users don't miss it, and consider adding a flag icon for those who don't understand the word "English".
- ‚úñÔ∏è **Clear Instructions:** The purpose of the page is clear, but some controls are not labelled, especially the most important: **email** and **password**. Add instruction labels like **Enter your email address** and **Create a password**.
- ‚úñÔ∏è **Caps Lock Indicator:** The page doesn't have one. This is especially dangerous, since there is neither a way to view the password nor a textbox to confirm it.
- ‚úÖ **Paste Allowed**
- ‚úñÔ∏è **Feedback Spinner:** No feedback is given when the <InlineUIElement>Sign Up</InlineUIElement> button is clicked. If you enable bandwidth throttling in browser tools, you can see how confusing this is. After attempting to sign up, the user can see no evidence that the button was clicked or that the service is processing anything. The page looks dead, and remains so for about five seconds ‚Äî easily enough time for a user to abandon registration and close the page.
- ‚úÖ **Long Timeouts:** By default, FusionAuth gives you 30 minutes to log in before the page times out. To adjust this, browse in the FusionAuth admin UI to <Breadcrumb>Tenants -> Edit your tenant -> Advanced tab</Breadcrumb> and change the value of <InlineField>Login timeout</InlineField>, then save the tenant.

Other attributes, like `autofocus` and `autocomplete`, are correctly set: `<input id="loginId" type="text" autocapitalize="none" autocomplete="on" autocorrect="off" spellcheck="false" autofocus="autofocus" placeholder="Email">`.
- ‚úÖ **`autocomplete`**
- ‚úñÔ∏è **`aria-hidden` ¬∑ `alt`:** No images on the page are purely decorative, so `aria-hidden` is not needed. But the screenshots do not have descriptions in `alt` attributes for screen readers (and for SEO and LLM understanding of your site).
- ‚úÖ **`aria-live`:** Since password validation is done only on form submission and not dynamically through JavaScript, no dynamic screen reader update is needed here.
- ‚úñÔ∏è **Standards Compliance:** To continuously test your page for HTML and CSS compliance, you can use a tool like the [Nu HTML Checker](https://github.com/validator/validator) locally. For a single test, use the [online version](https://validator.nu). As you can see below, the Iron Pixel theme contains numerous instances of invalid HTML and CSS that should be fixed.
  ![Nu HTML Checker](/img/docs/customize/look-and-feel/best-practices/nu.webp)
  FusionAuth handles unusual email addresses without a problem, including names with quotes, spaces, special characters, and Unicode text.
- ‚úÖ **Cross-Browser Compliance:** The site appears correct in Firefox and Chromium-based browsers, but thorough testing is very difficult. Safari runs only on expensive Apple hardware, and your local computer cannot install multiple versions of the same browser. If supporting as many browsers as possible is vital, you need to use a paid multi-browser testing service.
- ‚úÖ **Multiple Authentication:** Of the possible authentication methods (username, email, phone, social media, and passkey), FusionAuth supports logging in with a username, phone, or email (including email supplied by external authentication providers like Google). This example used email, and also provided social login through game services. Users appreciate this flexibility.
- ‚úÖ **Two-Factor Authentication:** The lack of two-factor authentication on this site is neither good nor bad. Presumably, a gaming site does not hold sensitive data and so chose ease of use over extra security.
- ‚úñÔ∏è **Custom Password Rules:** While not being overly strict with passwords makes life simpler for users, this site currently allows a password of `password`, which is not secure enough. If you want to avoid forcing the user to use special characters, consider increasing the password length requirement to 24 characters to encourage the user to enter a phrase.
  ![Password rules](/img/docs/customize/look-and-feel/best-practices/passwordRules.webp)
- ‚úñÔ∏è **CAPTCHA:** Since this site is just an example, it does not use CAPTCHA, which needs a paid provider and a paid FusionAuth license. However, in reality using CAPTCHA is unavoidable. If you don't have it, your site will quickly be overwhelmed with spam accounts or denial of service attacks from competitors. Be careful which provider you choose. Users frequently express frustration with Google's "Click all the bicycles" test when they fail for no apparent reason. Some users prefer hCAPTCHA's puzzles as straightforward, but some users find them even more onerous than Google's tests. While invisible CAPTCHA (which invisibly observes users' behavior on the page instead of showing a puzzle) seems like ideal usability, there is no remedy if it rejects a human user incorrectly.
  ![CAPTCHA](/img/docs/customize/look-and-feel/best-practices/captcha.webp)
- ‚úñÔ∏è **Rate-Limit Incorrect Passwords:** By default, FusionAuth doesn't rate limit anything. This can be configured easily in the <Breadcrumb>Security</Breadcrumb> tab of the tenant.
  ![Rate limit](/img/docs/customize/look-and-feel/best-practices/rateLimit.webp)

- ‚úÖ **Stay Logged In**
- ‚úÖ **Log Out**
- ‚úñÔ∏è **Matching Server/Client Validation:** As mentioned earlier, validation is not performed client-side at all in this design.

## Addendum: How To Create A Custom Advanced Theme From The FusionAuth Themes Templates

To create a custom theme (simple or advanced), first use the FusionAuth admin UI to duplicate the FusionAuth theme template in <Breadcrumb>Customizations -> Themes</Breadcrumb> and give it a UUID, for example `dc243231-0808-497a-9938-2d2a35860e01`.

Choose that theme in the dropdown of the <InlineUIElement>General</InlineUIElement> tab when editing the <Breadcrumb>Tenant</Breadcrumb> to apply it to all your applications.

Now you have your own theme to work with. To customize it, you need to download all the files to a local directory, edit them in a text editor, and re-upload them. This process is simplified by the FusionAuth Theme Helper tool. Run the commands below in a terminal to use it (you need Git and Node.js).

```sh
git clone https://github.com/FusionAuth/fusionauth-theme-helper
cd fusionauth-theme-helper
cp .env.sample .env
vi .env # set your FusionAuth key from Settings -> API Keys

# Example:
#API_KEY=33052c8a-c283-4e96-9d2a-eb1215c69f8f-not-for-prod
#THEME_ID=dc243231-0808-497a-9938-2d2a35860e01
#FUSIONAUTH_URL=http://localhost:9011
#TMP_DIR=tmp

# save and exit vi

npm install
./download.sh
```

You should now have a copy of the custom theme files in the `fusionauth-theme-helper/tmp` directory.

Once you've made your changes, run `./upload.sh` to save your changes back to FusionAuth so you can see what they look like.

If you want to use Iron Pixel as a template, download the files from https://github.com/FusionAuth/fusionauth-themes/tree/main/packages/fusionauth/iron-pixel. You should have a directory containing `input.css` and a `tpl` directory ("TemPLate"). Copy the contents of `tpl` up to the same level as `input.css`. Then copy all the files into the `tmp` folder that the FusionAuth Theme Helper download script created. You now have a copy of Iron Pixel, ready for editing and uploading.
