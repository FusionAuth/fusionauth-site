## Self-Service SSO in Tenant Manager

For the first time, tenant administrators can configure SSO entirely within [Tenant Manager](/docs/lifecycle/manage-users/tenant-manager).

First, ask your FusionAuth admin to enable Self-Service SSO in the FusionAuth Admin UI -- they'll only have to do this once:

![Complete Registration Settings](/img/docs/lifecycle/register-users/complete-registration-settings.png)

From there, tenant administrators can configure, test, and enable SSO in Tenant Manager using the <Breadcrumb>Single Sign-On</Breadcrumb> page:

![Edit identity provider](/img/docs/lifecycle/manage-users/tenant-manager/edit-idp.png)

## Identity Provider Testing API

To support self-service SSO in Tenant Manager, we created a whole new [API for testing identity provider configurations](/docs/apis/identity-providers/test), and a UI in Tenant Manager for visual testing. Tenant administrators can use these tests to verify an identity provider configuration before enabling an identity provider.

To test an identity provider, open the <Breadcrumb>Single Sign-On</Breadcrumb> page, expand the action menu for an identity provider, and click <InlineUIElement>Test</InlineUIElement>.

The connection test start page shows a summary of your configuration.

Click <InlineUIElement>Start Test</InlineUIElement> to begin the test.

![Connection Test Start View](/img/docs/lifecycle/manage-users/tenant-manager/test-start.png)

Complete the login flow using an identity provider, and you'll see the test success page:

![Connection Test Completion View](/img/docs/lifecycle/manage-users/tenant-manager/test-completion.png)

If your login flow works, you can click <InlineUIElement>Enable for production</InlineUIElement> to ship SSO to your users.

If your login flow doesn't work, you'll see the test failure page, which provides logs and other information you can use to debug the problems with your configuration. Keep testing until your configuration works.

## Declarative User Reconciliation

Many of our users currently use reconcile lambdas like our [OIDC reconcile lambda](/docs/extend/code/lambdas/reconcile/openid-connect-response-reconcile) to reconcile user identities across identity providers. But it turns out we didn't need anything as powerful as a lambda to define reconciliation for self-service SSO in Tenant Manager.

We just needed [attribute mappings](/docs/lifecycle/authenticate-users/identity-providers/#attribute-mappings). In fact, to use Tenant Manager SSO, you need to configure attribute mappings -- a reconcile lambda just won't do.

Attribute mappings allow you to map claims from an Identity Provider response directly to FusionAuth user and registration fields. FusionAuth admins can configure these key-value pairs using the [Identity Provider API](/docs/apis/identity-providers/) or the Identity Provider configuration page in the Admin UI. Tenant admins can customize them for their tenants in [Tenant Manager](/docs/lifecycle/manage-users/tenant-manager#attribute-mappings).

During reconciliation, FusionAuth reads claim values using the attribute mapping values declared in the Identity Provider response. FusionAuth assigns the retrieved values to the corresponding key on the FusionAuth user or registration. But don't worry, all attribute mappings are optional; if FusionAuth can't retrieve a claim value or set a key, it omits the mapping, but reconciliation continues.

## Self-Service Incomplete User Registration

This release also adds a new [self-service option for applications](/docs/lifecycle/register-users/complete-registration) that allows users to complete an existing (but incomplete) registration, but does not allow entirely new users to create a new registration. We call it **complete registration**.

![Complete Registration](/img/docs/get-started/core-concepts/application-complete-registration-self-service-basic.png)

When complete registration is enabled on an application, FusionAuth checks whether a registered user lacks any user or registration data required by the configured form. After authenticating, the user must supply that data before completing the hosted login flow.

To enable complete registration for an application:

1. Navigate to the <Breadcrumb>Applications</Breadcrumb> tab.
1. <InlineUIElement>Edit</InlineUIElement> the application you want to configure.
1. Navigate to the <Breadcrumb>Registration</Breadcrumb> tab.
1. Scroll to the <InlineUIElement>Self-service registration</InlineUIElement> section.
1. Set the <InlineUIElement>Registration mode</InlineUIElement> to <InlineField>Complete registration</InlineField>.
1. Configure the [basic](/docs/lifecycle/register-users/basic-registration-forms) or [advanced](/docs/lifecycle/register-users/advanced-registration-forms) registration form.
1. <InlineUIElement>Save</InlineUIElement> the application.

## Filter Logs by Tenant

Finally, we added the `auditLog.tenantId` field to the [Audit Logs API](/docs/apis/audit-logs). Now you can filter audit logs by tenant using the API and in [Tenant Manager](/docs/lifecycle/manage-users/tenant-manager). This should make it that much easier to debug problems that only impact a single tenant.
