The Model Context Protocol (MCP) is rapidly becoming the standardized API for large language models (LLMs) and AI agents to interact with the real world—calling APIs, accessing databases, and executing business logic. Like the early days of REST, builders are moving fast, often prioritizing functionality over security.

We’re already seeing the security mistakes we made a decade ago showing up with this new technology. 

{/* more */}

You can see some of these in the MCP [security advisory list](https://github.com/modelcontextprotocol/servers/security/advisories) which lists path traversal, path validation and injection issues. But it's not just in the core software, it's also in implementation. Whether it is [broken tenant isolation](https://www.bleepingcomputer.com/news/security/asana-warns-mcp-ai-feature-exposed-customer-data-to-other-orgs/), [networking misconfiguration](https://composio.dev/content/mcp-vulnerabilities-every-developer-should-know), or [tool poisoning](https://invariantlabs.ai/blog/whatsapp-mcp-exploited), you want to avoid the negative impact. The pattern is familiar, but the stakes are new. AI agents can act autonomously, which means a misconfigured MCP server allows more damage than a misconfigured REST endpoint.

One way to do this is to require authentication and authorization using an Authorization Server (commonly abbreviated AS). If your MCP server is providing access on the internet to anything confidential or if the data and functionality it offers varies depending on the user making the request, then lock it down. An unauthenticated remote server is fantastic for public documentation (the [FusionAuth docs MCP server](/docs/get-started/download-and-install/development/docs-mcp-server), for example) or non-sensitive, shared resources, but anything beyond that needs a proper security layer.

Now, the MCP specification itself does not enforce security at the protocol level. This is a deliberate choice by the authors, but "left to the implementer" often translates to "skipped until a breach forces re-evaluation". Don't let that be you.

Luckily, MCP does specify a way for you to secure your remote server. It relies on OAuth and the ecosystem built up around that. So, not all of the mistakes from the API era are showing up.

By delegating authentication to an AS, your system verifies who the user is and what their permissions are, all before the request hits your server's core logic. This is done through the OAuth Authorization Code grant, using well-known concepts like scopes and consents. While there's an [MCP extension which allows for client credentials access](https://modelcontextprotocol.io/extensions/auth/oauth-client-credentials), this post is focused on the more typical "on-behalf-of" flow. This is called that because the user explicitly grants the MCP client access to the MCP server on their behalf.

Here is the flow that gets an MCP client authenticated and authorized to use your protected remote MCP server:

* server discovery
* client registration
* user authentication and granting scopes
* access token issuance and validation

Let's take a look at each of these.

### Server Discovery

First, the MCP client (like Claude Desktop or an AI agent) needs to be configured with the location of your MCP server. How this is done depends on your MCP client, but can range from editing a JSON file to adding a URL to a form.

When the client first hits your MCP server, the server says, “I don’t know you. Go here to get me credentials”.

This response sends the client to an AS the MCP server trusts.

### Client Registration

When the MCP client gets to the AS, the AS needs to know some things about it.

There are a few ways for this to happen. In the [order recommended by the spec](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization#client-registration-approaches):

* the MCP client can be pre-registered with the AS. This means that some other process created a link between the MCP client and AS. This is referred to as "out of band" but really means "we don't know and we don't care how it happened, as long as it happened". This is similar to showing up to a formal gala party and flashing your invitation. You've been invited and are known before you talk to the person at the door.
* the MCP client can register at the time it first interacts with the AS. There are two technical paths, [Client ID Metadata Documents (CIMD)](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-client-id-metadata-document-00) and [Dynamic Client Registration (DCR)](https://datatracker.ietf.org/doc/html/rfc7591). The former is newer but seems to be the future, with the latter being explicitly retained for backwards compatibility. This is similar to a bouncer checking a driver's license at a bar. As long as you meet certain criteria, you're in.
* prompt the user behind the MCP client to enter credentials. I haven't seen this in the wild and am not quite sure how it would work. This is like showing up to a party without an invitation and having the host come to the door to personally vouch for you.

The client registration process can offer additional security at the cost of additional friction. The pre-registration process means that you'll have some kind of system awareness of the MCP clients interacting with your server. CIMD and DCR both allow for locking down which MCP clients can access an AS (and, by extension, an MCP server).

In all cases, the user will also be authenticated after the client has been registered. Let's see what that looks like.

### User Authentication

After the MCP client is recognized by the AS, the user is authenticated at the AS. This process can take whatever form satisfies the AS, in terms of the method of authentication and whether or not additional factors like a TOTP code are required.

As part of the process, the user is typically presented with a consent screen asking for the [OAuth scopes](/blog/how-to-design-oauth-scopes) that the MCP client needs. How this maps to the functionality the MCP server controls varies, but you can map a scope to a set of tools the MCP client can access. So you might have a scope of `account:read` which could map to the following endpoints and methods:

* `GET /api/v1/account`: retrieve the authenticated user's full account profile
* `GET /api/v1/account/usage`: fetch usage metrics for an account 
* `GET /api/v1/account/billing`: read billing details
* `GET /api/v1/account/members`: list the users/members associated with the account, if any
* `GET /api/v1/account/preferences`: retrieve account-level preferences and config settings

A scope of `account:write` might on the other hand, grant access to endpoints and methods like `PUT /api/v1/account` or `POST /api/v1/account/members`.

You can also include more standard scopes like `offline_access` for refresh tokens and OIDC scopes such as `profile`.

Scopes can be required or optional, but the user should always have the option to cancel the authentication if they feel the permissions requested by the MCP client are too broad.

Next up, the access token is created.

### Access Token Issuance and Validation

Upon successful login and consent, the AS issues a time-bound OAuth access token to the client. The MCP client then presents this token to the protected MCP server. One of the security benefits of MCP when compared with other agentic tooling is that the MCP client harness holds the token and it is at no time shared with the LLM. This means that an attacker can't trick an LLM into directly revealing the credential, since it does not have it.

When it receives the token, the MCP server must perform strict validation on this token, checking critical fields like the audience, expiration time, issuer, and the allowed scopes. If AS issued a JWT for the token the MCP server should also verify the signature and follow [other JWT best practices too](/articles/tokens/building-a-secure-jwt).

If the token is valid, the MCP server proceeds, executing the requested call based on the verified identity.

## What's Next?

MCP is moving fast, and the security mistakes of the REST API era are already showing up in the wild. The good news is that the MCP specification doesn't leave you to invent a security model from scratch. It leans on OAuth, a battle-tested standard with years of expert scrutiny and a mature ecosystem behind it.

By delegating authentication and authorization to an AS, you get verified identities, user-controlled consent, and time-limited access tokens, before a single tool call executes. That's a security baseline that most ad-hoc API implementations don't have.

A few principles to carry forward as you build:
* Keep scopes narrow. An MCP client requesting broad permissions is asking for the AI equivalent of `root` with all the blast radius and none of the judgment. Design your scopes around the minimum access each client actually needs.
* Don't stop at the MCP boundary. The specification secures the client-to-server handshake, but your MCP server likely calls downstream services of its own. You are responsible for securing those connections too. Do not forward the user's access token downstream; issue purpose-specific credentials instead.
* Treat unauthenticated servers as a deliberate choice. An open, unauthenticated MCP server is appropriate for public resources like documentation. Anything involving user data, business logic, or sensitive operations needs a proper security layer. 

The tools and infrastructure are here. The specification is clear. The only variable is whether you use these to secure your MCP server before something goes wrong.

Or after.
